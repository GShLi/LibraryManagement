## 1. 架构概述

### 1.1 架构目标

1. **前后端分离**：前端 SPA 与后端 RESTful API 完全解耦，独立开发、独立部署、独立扩展
2. **分层架构**：后端采用 Controller → Service → DAO 三层架构，职责清晰，便于维护和测试
3. **轻量化部署**：单体应用 + 嵌入式数据库，零外部依赖，适合小型图书馆/机构图书室场景
4. **RBAC 权限模型**：基于角色的访问控制，前端路由守卫 + 后端中间件双重校验

### 1.2 架构选型理由

采用**前后端分离 + 后端分层**的架构模式，理由如下：

- **项目规模**：中小型图书管理系统，业务复杂度适中，分层架构既能保证代码结构清晰，又不会引入微服务架构的额外复杂度
- **团队适配**：Vue3 + Express 技术栈学习曲线平缓，中小团队可快速上手
- **部署约束**：单机部署需求，SQLite 作为嵌入式数据库无需独立数据库服务，降低运维成本
- **扩展预留**：分层架构下，未来如需切换到 PostgreSQL/MySQL 或拆分为微服务，只需替换 DAO 层或拆分 Service 层即可

### 2.2 分层职责

| 层次 | 组件 | 职责 | 约束 |
|------|------|------|------|
| 客户端层 | Vue3 SPA | 界面渲染、用户交互、前端路由、状态管理、请求拦截 | 不包含业务逻辑，不做权限决策（仅做 UI 隐藏） |
| 中间件层 | Express Middleware | 认证校验、权限校验、请求日志、CORS、参数校验、统一错误处理 | 不做业务处理，异常统一返回标准错误格式 |
| 控制器层 | Express Router | 接收请求、参数提取、调用 Service、封装响应 | 不做数据库直接操作，不包含业务逻辑 |
| 业务逻辑层 | Service 模块 | 核心业务逻辑、事务管理、数据校验、业务规则 | 不处理 HTTP 请求/响应，不直接操作 ORM（通过 DAO 辅助函数） |
| 数据访问层 | Sequelize ORM | 数据持久化、模型定义、关联查询、事务支持 | 不包含业务逻辑 |
| 数据层 | SQLite 3 | 数据存储、索引、备份 | 单文件数据库，WAL 模式 |

### 2.3 关键设计决策

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 前端路由模式 | History 模式 | URL 美观，无 `#` 符号，需后端配合 nginx try_files |
| API 版本管理 | URL 前缀 `/api/v1/` | 预留版本升级能力，当前统一使用 v1 |
| 认证方案 | JWT (无状态) | 无需服务端 Session 存储，适合单机部署，Token 含角色信息 |
| 密码存储 | bcrypt (cost=10) | 行业标准，抗暴力破解 |
| 日志方案 | Winston (文件 + 控制台) | 结构化日志，按日切割，保留 30 天 |
| 文件上传 | multer 中间件 | 轻量级，支持封面图片上传到本地磁盘 |
| 定时任务 | node-cron | 轻量级，在 Express 进程内执行每日逾期检测 |
| 跨域方案 | cors 中间件 | 开发环境允许前端 dev server 跨域，生产环境同源部署无跨域 |

## 3. 技术选型

### 3.1 技术栈总览

| 层级 | 技术 | 版本 | 选型理由 |
|------|------|------|---------|
| **前端框架** | Vue 3 | ≥ 3.3 | 渐进式框架，组合式 API 提升逻辑复用；生态成熟，社区活跃；与 Element Plus 无缝集成 |
| **UI 组件库** | Element Plus | ≥ 2.3 | 企业级 Vue 3 组件库，提供表格/表单/弹窗/导航等完整后台组件；中文文档完善；支持响应式 |
| **前端构建** | Vite | ≥ 4.0 | 开发时极速冷启动（ESM），HMR 热更新；生产构建基于 Rollup，Tree Shaking 优化好 |
| **前端状态管理** | Pinia | ≥ 2.0 | Vue 3 官方推荐，TypeScript 友好，模块化设计，DevTools 支持 |
| **前端路由** | Vue Router | ≥ 4.0 | Vue 3 官方路由，支持路由守卫、懒加载、动态路由 |
| **HTTP 客户端** | Axios | ≥ 1.0 | 拦截器机制统一处理 Token 注入和 401 响应；请求/响应转换 |
| **图表** | ECharts | ≥ 5.0 | 国产开源，文档完善，支持仪表盘所需折线图/柱状图/饼图 |
| **后端框架** | Express | ≥ 4.18 | Node.js 生态最成熟的 Web 框架；中间件机制灵活；学习成本低 |
| **ORM** | Sequelize | ≥ 6.0 | 支持 SQLite 方言；提供模型定义、关联查询、事务、迁移等完整能力 |
| **认证** | jsonwebtoken | ≥ 9.0 | JWT 标准实现，支持 RS256/HS256，Payload 承载角色信息 |
| **密码加密** | bcryptjs | ≥ 2.0 | 纯 JS 实现，无原生编译依赖，跨平台部署方便 |
| **参数校验** | express-validator | ≥ 7.0 | 链式校验 API，与 Express 中间件模式完美契合 |
| **日志** | winston | ≥ 3.0 | Node.js 标准日志库，支持多 Transport（文件/控制台），按级别过滤 |
| **定时任务** | node-cron | ≥ 3.0 | Cron 表达式驱动，进程内轻量调度，适合单机逾期检测任务 |
| **文件上传** | multer | ≥ 1.0 | Express 生态标准文件上传中间件 |
| **数据库** | SQLite 3 | ≥ 3.35 | 零配置、零运维、嵌入式；WAL 模式支持并发读；单文件备份简单；满足 10 万级数据需求 |
| **运行时** | Node.js | ≥ 18 LTS | 长周期维护保障，ES2022+ 特性原生支持 |

### 3.2 技术选型详解

#### 3.2.1 前端：Vue 3 + Element Plus + Vite

- **Vue 3** 的组合式 API（Composition API）允许将同一功能逻辑聚合在一起，比选项式 API 更适合中大型表单和复杂交互场景（如借书流程的多步骤操作）
- **Element Plus** 提供了桌面端后台管理系统所需的全套组件：Table（表格）、Form（表单）、Dialog（弹窗）、Menu（导航）、Tag（状态标签），无需从零构建
- **Vite** 开发服务器基于原生 ESM，大项目冷启动在毫秒级，对比 Webpack 的几十秒显著改善开发体验

#### 3.2.2 后端：Express + Sequelize + SQLite

- **Express** 是最广泛使用的 Node.js Web 框架，中间件生态完善（cors、helmet、morgan 等），对于本项目的 API 数量（约 40 个端点）完全胜任
- **Sequelize** 是 Node.js 社区最成熟的 ORM，支持 SQLite 方言。通过 Model 定义即可完成建表和迁移，避免手写 SQL DDL。其事务 API 支持借还书等需要原子性保障的场景
- **SQLite** 的选择基于以下考虑：
  - 项目定位为中小型图书馆，数据量在百万级以内，SQLite 完全满足
  - 单机部署，无需独立的数据库进程，降低运维复杂度
  - 备份简单：直接复制 .db 文件即可
  - WAL（Write-Ahead Logging）模式下支持一写多读并发
  - 零配置，即开即用

#### 3.2.3 未选用方案说明

| 候选技术 | 不选用的理由 |
|---------|------------|
| React + Ant Design | 团队更熟悉 Vue 生态；Element Plus 对表单/表格场景的开箱支持更好 |
| PostgreSQL / MySQL | 需要独立安装和运维数据库服务，不符合"轻量化部署"的目标 |
| TypeORM / Prisma | Sequelize 对 SQLite 的支持更成熟稳定；Prisma 需要额外生成引擎 |
| NestJS | 对于本项目规模过度设计；Express 的直接性更适合中小项目 |
| PM2 进程管理 | 可用但不必须；单机部署可直接用 systemd 管理进程，PM2 为可选增强 |

#### 后端模块职责矩阵

| 模块 | 控制器层职责 | 业务逻辑层职责 |
|------|------------|-------------|

| book | 图书检索、入库、编目、下架、重新上架、批量导入 | ISBN 校验、条码生成、副本批量创建、下架前置校验（副本状态） |

| stats | 借阅排行、逾期统计、概览统计 | 聚合查询、分组统计 |

### 7.3 安全措施汇总

| 安全维度 | 措施 | 实现位置 |
|---------|------|---------|
| 传输安全 | 生产环境 HTTPS，开发环境 HTTP | Nginx / 网络层 |
| 认证 | JWT Bearer Token，24h 有效期 | auth middleware |
| 密码安全 | bcrypt (cost=10) 加盐哈希 | authService |

| 输入校验 | 前后端双重校验：前端 el-form rules，后端 express-validator | validator middleware |
| SQL 注入防护 | Sequelize ORM 参数化查询 | DAO 层 |
| XSS 防护 | Vue 默认转义 + helmet 中间件 | 前端框架 + Express |
| CSRF 防护 | JWT Bearer Token 机制天然防护（非 Cookie） | 认证方案 |
| 暴力破解防护 | 连续失败 5 次锁定 15 分钟 | authService |
| 审计追踪 | 关键操作记录 audit_log，不可删除 | auditLog model |
| 数据备份 | 每日自动备份 SQLite 文件 + 手动备份 | overdueCheck job + systemService |
| 错误信息 | 生产环境不暴露内部错误详情 | errorHandler middleware |

## 8. 数据架构

### 8.1 数据库选型

SQLite 3.35+，启用 WAL（Write-Ahead Logging）模式：

- **WAL 模式优势**：读写不互斥，支持一写多读并发，显著提升借还操作高频场景下的并发能力
- **配置**：`PRAGMA journal_mode=WAL; PRAGMA foreign_keys=ON;`
- **备份策略**：每日凌晨使用 `VACUUM INTO 'backups/library_YYYYMMDD.db'` 进行热备份

### 8.2 实体关系

详见《数据库设计文档》，核心实体包括 9 张表：

| 表 | 说明 | 预估容量 |
|----|------|---------|
| users | 用户账号 | 5 万 |
| readers | 读者档案 | 5 万 |
| books | 书目信息 | 10 万 |
| book_copy | 图书副本 | 15 万 |
| borrow | 借阅记录 | 100 万 |
| fine | 罚款记录 | 10 万 |
| reserve | 预约记录 | 5 万 |
| audit_log | 操作日志 | 200 万 |
| system_config | 系统配置 | ~20 条 |

### 8.3 数据流架构

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   前端 SPA   │────▶│  Express API │────▶│  SQLite DB  │
│  (只读渲染)  │◀────│  (读写分离)  │◀────│  (WAL 模式)  │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                        ┌──────┴──────┐
                                        │  定时备份    │
                                        │  (cron)     │
                                        └──────┬──────┘
                                               │
                                        ┌──────┴──────┐
                                        │  backups/   │
                                        │  (最多7份)   │
                                        └─────────────┘
```

## 9. 非功能性架构设计

### 9.1 性能设计

| 指标 | 目标 | 设计方案 |
|------|------|---------|
| 接口响应时间 | 普通查询 ≤ 500ms，统计 ≤ 2s | Sequelize 索引优化；统计查询走聚合 SQL；前端列表懒加载 |
| 并发能力 | 50 用户同时在线 | SQLite WAL 模式支持并发读；Express 事件循环模型天然异步非阻塞 |
| 首屏加载 | ≤ 2s (3G) | Vite 代码分割 + 路由懒加载 + gzip 压缩 + 静态资源缓存 |
| 批量导入 | 5000 条 ≤ 30s | Sequelize bulkCreate + 事务批量提交 |

### 9.2 可用性设计

| 指标 | 目标 | 设计方案 |
|------|------|---------|
| 系统可用率 | 99.5% (月停机 ≤ 2h) | systemd 进程守护 + `Restart=on-failure` |
| 故障恢复 RTO | ≤ 4h | 备份恢复脚本 + 部署文档 |
| 数据一致性 | 事务 ACID | Sequelize 事务 API 包裹多表操作 |
| 优雅降级 | 前端错误提示 | Axios 拦截器统一处理网络异常；Vue errorBoundary 组件级容错 |

### 9.3 可扩展性设计

- **水平扩展预留**：当前为单机部署，但分层架构使未来拆分变得简单
  - 前端已独立，可直接部署到 CDN
  - 后端 Service 层独立，未来可按业务域拆分为微服务
  - DAO 层通过 ORM 抽象，切换到 PostgreSQL 仅需修改 Sequelize 方言配置
- **API 版本化**：通过 `/api/v1/` 前缀预留版本管理空间
- **配置外置**：系统配置存储在 system_config 表，运行时动态读取，无需重启

## 10. 技术风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| SQLite 并发写入瓶颈 | 高并发借还时性能下降 | WAL 模式；单机场景 50 并发用户足够；监控慢查询 |
| SQLite 单点故障 | 数据丢失 | 每日自动备份 + 手动备份；备份保留 7 天 |
| Node.js 单进程阻塞 | 服务不可用 | systemd 自动重启；避免同步阻塞操作；CPU 密集任务拆分 |
| JWT 无状态无法撤销 | 安全风险 | Token 有效期 24h；密码修改后旧 Token 自然过期 |
| 前端状态丢失 | 用户刷新后数据丢失 | 关键状态（Token）存储在 localStorage；临时状态（借书流程）保持在 Pinia |
