## 1. 概述

### 1.1 设计目标

本文档定义图书管理系统的非功能性设计方案，覆盖性能、安全性、可用性、可扩展性、可维护性五个维度，为开发实现和运维部署提供具体可落地的技术指导。

### 1.2 量化指标总览

| 维度 | 指标 | 目标值 |
|------|------|--------|
| 性能 | 首屏加载时间 (P95) | ≤ 2s (3G 网络) |
| 性能 | API 响应时间 (P95) | 普通查询 ≤ 500ms，复杂统计 ≤ 2s |
| 性能 | 并发用户数 | 50 用户同时在线操作 |

## 2. 性能设计

### 2.1 前端性能优化

#### 2.1.1 路由懒加载与代码分割

Vue Router 配合 Vite 的动态 `import()` 实现按路由分割代码块，确保用户仅加载当前页面所需的 JS/CSS：

```
// 管理后台路由均采用懒加载
const AdminDashboard = () => import('@/views/admin/Dashboard.vue')
const BookList       = () => import('@/views/admin/books/BookList.vue')
const BookAdd        = () => import('@/views/admin/books/BookAdd.vue')
const BorrowPage     = () => import('@/views/admin/borrows/BorrowPage.vue')
const ReturnPage     = () => import('@/views/admin/borrows/ReturnPage.vue')
const ReaderList     = () => import('@/views/admin/readers/ReaderList.vue')
// ...其余路由同理
```

期望效果：
- 初始加载仅加载登录页 + 公共框架代码（预估 < 200KB gzip）
- 每个管理页面独立 chunk，按需加载

#### 2.1.2 组件异步加载

对于非首屏必需的重组件（ECharts 图表、富文本编辑器），采用异步组件：

```
import { defineAsyncComponent } from 'vue'
const BarChart = defineAsyncComponent(() => import('@/components/BarChart.vue'))
```

#### 2.1.3 静态资源优化

| 策略 | 实现方式 | 预期效果 |
|------|---------|---------|
| 代码压缩 | Vite 生产构建默认 terser 压缩 + CSS 压缩 | JS/CSS 体积减少 60%+ |
| Gzip 压缩 | Nginx `gzip on`，压缩 JS/CSS/HTML/JSON | 传输体积减少 70%+ |
| 图片懒加载 | `<img loading="lazy">` + Element Plus `el-image` lazy 属性 | 减少首屏图片请求 |
| 封面图压缩 | 上传时 multer 限制 500KB，前端显示时使用缩略图 | 减少封面图带宽 |
| 字体图标 | Element Plus 内置图标，不引入额外字体文件 | 减少网络请求 |
| 缓存策略 | Nginx 对 `/assets/*` 设置 `Cache-Control: max-age=31536000`（文件名含 hash） | 二次访问零请求 |

#### 2.1.4 前端请求优化

| 策略 | 实现 | 说明 |
|------|------|------|
| 防抖搜索 | `useSearch.js` composable，300ms 防抖 | 避免每次按键触发 API 请求 |
| 列表分页 | 每页 20 条，后端分页返回 | 减少单次数据传输量 |
| 请求取消 | Axios CancelToken 取消重复请求 | 用户快速切换筛选条件时取消前一次未完成的请求 |
| 预加载 | 图书详情页链接 hover 时 prefetch | 提升详情页打开速度 |

### 2.2 后端性能优化

#### 2.2.1 数据库索引策略

基于数据库设计文档中定义的索引方案，确保以下关键查询路径有索引覆盖：

| 查询场景 | 索引 | 索引类型 |
|---------|------|---------|
| 用户登录 (username) | `idx_users_username` | UNIQUE |
| 读者证号/手机号查询 | `idx_readers_reader_no` / `idx_readers_phone` | UNIQUE |
| 图书检索 (书名模糊) | `idx_books_title` | NORMAL |
| 图书检索 (作者模糊) | `idx_books_author` | NORMAL |
| 图书检索 (分类筛选) | `idx_books_category` | NORMAL |
| 条码精确查询 | `idx_copy_barcode` | UNIQUE |
| 借阅历史 (按读者) | `idx_borrow_reader` | NORMAL |
| 逾期扫描 (日期范围) | `idx_borrow_due_date` + `idx_borrow_status` | NORMAL |
| 操作日志 (时间范围) | `idx_log_created` | NORMAL |

SQLite 查询优化配置：

```sql
PRAGMA journal_mode = WAL;        -- 读写并发，不互斥
PRAGMA synchronous = NORMAL;      -- 平衡安全性与写入性能
PRAGMA cache_size = -64000;       -- 64MB 页面缓存
PRAGMA mmap_size = 268435456;     -- 256MB 内存映射 I/O
PRAGMA foreign_keys = ON;         -- 强制外键约束
PRAGMA temp_store = MEMORY;       -- 临时表在内存中
```

#### 2.2.2 查询优化

| 策略 | 说明 |
|------|------|
| 分页查询 | 所有列表接口强制分页（默认 page=1, pageSize=20），避免全表扫描 |
| 冗余计数 | `books.total_copies` / `books.available_copies` / `readers.current_borrowed` 作为冗余字段，避免列表查询时频繁 JOIN + COUNT |
| 选择性 JOIN | 列表查询仅 JOIN 必要表；详情查询才加载完整关联 |
| 统计查询 | 聚合查询（借阅排行、逾期统计）使用 SQL GROUP BY + 聚合函数，不在应用层做循环统计 |
| 慢查询监控 | Winston 日志记录 API 响应时间，超过 1s 的请求标记 warn 级别 |

#### 2.2.3 Sequelize 查询最佳实践

```js
// 分页查询 — 使用 offset + limit
const { rows, count } = await Book.findAndCountAll({
  where: conditions,
  offset: (page - 1) * pageSize,
  limit: pageSize,
  order: [['created_at', 'DESC']],
  // 列表查询不加载关联数据
  include: [],
  // 只查询需要的字段
  attributes: ['id', 'isbn', 'title', 'author', 'status', 'available_copies']
})

// 统计查询 — 使用聚合
const ranking = await Borrow.findAll({
  attributes: [
    'copy_id',
    [sequelize.fn('COUNT', sequelize.col('copy_id')), 'borrow_count']
  ],
  where: { borrow_date: { [Op.between]: [start, end] } },
  group: ['copy_id'],
  order: [[sequelize.literal('borrow_count'), 'DESC']],
  limit: 20
})
```

#### 2.2.4 事务优化

借书、还书等关键操作使用 Sequelize 事务，事务范围最小化：

```js
const result = await sequelize.transaction(async (t) => {
  // 仅在事务内执行必须原子化的操作
  const borrow = await Borrow.create({ ... }, { transaction: t })
  await BookCopy.update({ status: 'borrowed' }, { where: { id: copyId }, transaction: t })
  await Reader.increment('current_borrowed', { where: { user_id: readerId }, transaction: t })
  return borrow
})
// 日志记录在事务成功后异步写入，不在事务内
await AuditLog.create({ ... })
```

#### 2.2.5 Node.js 运行时优化

| 策略 | 说明 |
|------|------|
| 异步非阻塞 | 所有 I/O 操作使用 async/await，避免同步阻塞（不使用 `fs.readFileSync` 等） |
| 连接池 | Sequelize 连接池：`pool: { max: 10, min: 2, acquire: 30000, idle: 10000 }` |
| 内存控制 | Node.js 启动参数 `--max-old-space-size=512` 限制堆内存上限 |

## 3. 安全性设计

### 3.1 认证方案

#### 3.1.1 JWT Token 设计

```
Token 结构（Payload）:
{
  "sub": 1001,                     // 用户 ID
  "role": "librarian",             // 角色
  "iat": 1712345678,               // 签发时间
  "exp": 1712432078                // 过期时间（24h 或 7d）
}
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 签名算法 | HS256 | 对称签名，单机部署足够 |
| 密钥来源 | 环境变量 `JWT_SECRET` | `.env` 文件，不提交到代码仓库 |
| 默认有效期 | 24h | 不勾选"记住我" |
| 记住我有效期 | 7d | 勾选"记住我" |
| Token 存储 | 前端 `localStorage` | 不存储在 Cookie，规避 CSRF |

#### 3.1.2 认证中间件流程

```
请求到达 → auth middleware
  ├─ 检查 Authorization Header 是否存在
  │   └─ 否 → 401 "请先登录"
  ├─ 解析 Bearer Token → jwt.verify(token, JWT_SECRET)
  │   ├─ Token 无效/签名错误 → 401 "认证失败"
  │   ├─ Token 过期 → 401 "登录已过期，请重新登录"
  │   └─ Token 有效 → 提取 { id, role } → req.user
  ├─ 查询数据库校验用户状态
  │   ├─ 用户不存在/已删除 → 401 "用户不存在"
  │   └─ 用户已禁用 → 403 "账号已被禁用"
  └─ next() → 后续中间件/路由
```

#### 3.1.3 登录安全策略

| 策略 | 实现 |
|------|------|
| 密码加密 | bcryptjs, cost=10, 加盐哈希 |
| 暴力破解防护 | 同一账号连续失败 5 次，锁定 15 分钟（`locked_until` 字段） |
| 失败计数重置 | 登录成功后 `login_attempts` 归零 |
| 统一错误提示 | 用户名不存在/密码错误统一返回"用户名或密码错误"，防止用户名枚举 |
| 密码强度 | 注册时强制 8-32 位，必须含字母和数字 |

| 位置 | 实现 | 目的 |

### 3.3 输入校验

#### 3.3.1 前后端双重校验

| 层 | 工具 | 校验内容 |
|----|------|---------|
| 前端 | Element Plus `el-form` rules | 必填校验、格式校验（手机号、ISBN、邮箱）、长度限制 |
| 后端 | `express-validator` 中间件 | 类型校验、格式正则、范围校验、SQL 注入/XSS 防护 |

后端校验示例：

```js
// 注册接口校验链
const validateRegister = [
  body('username')
    .trim().isLength({ min: 3, max: 50 }).withMessage('用户名长度 3-50 位')
    .matches(/^[a-zA-Z0-9_]+$/).withMessage('用户名只能包含字母、数字和下划线'),
  body('password')
    .isLength({ min: 8, max: 32 }).withMessage('密码长度 8-32 位')
    .matches(/^(?=.*[a-zA-Z])(?=.*\d)/).withMessage('密码必须包含字母和数字'),
  body('phone')
    .matches(/^1\d{10}$/).withMessage('请输入正确的手机号'),
  body('name')
    .trim().isLength({ min: 1, max: 50 }).withMessage('姓名不能为空'),
]
```

## 4. 可用性设计

### 4.1 错误处理策略

#### 4.1.1 后端全局错误处理

Express 全局错误处理中间件，捕获所有未处理异常，返回统一格式错误响应：

```js
// middleware/errorHandler.js
function errorHandler(err, req, res, next) {
  // Winston 记录完整错误堆栈
  logger.error({ message: err.message, stack: err.stack, url: req.url, method: req.method })

  // 区分已知业务错误和未知系统错误
  if (err.isBusinessError) {
    return res.status(err.statusCode || 400).json({
      code: err.code,
      message: err.message,      // 业务错误返回用户可理解的描述
      data: null
    })
  }

  // Sequelize 校验错误
  if (err.name === 'SequelizeValidationError') {
    const messages = err.errors.map(e => e.message)
    return res.status(400).json({ code: 400, message: messages.join('; '), data: null })
  }

  // 未知错误：不暴露内部信息
  const isProduction = process.env.NODE_ENV === 'production'
  return res.status(500).json({
    code: 500,
    message: isProduction ? '服务器内部错误，请稍后重试' : err.message,
    data: null
  })
}
```

#### 4.1.2 统一响应格式

成功响应：
```json
{ "code": 200, "message": "ok", "data": { ... } }
```

分页响应：
```json
{
  "code": 200,
  "message": "ok",
  "data": {
    "list": [...],
    "total": 1000,
    "page": 1,
    "pageSize": 20
  }
}
```

错误响应：
```json
{ "code": 400, "message": "描述错误原因的中文信息", "data": null }
```

#### 4.1.3 业务错误分类

| 错误类型 | HTTP 状态码 | 使用场景 |
|---------|-----------|---------|
| 参数校验失败 | 400 | 前端表单校验未通过 |
| 认证失败 | 401 | Token 缺失/无效/过期 |
| 权限不足 | 403 | 用户角色无权访问该资源 |
| 资源不存在 | 404 | 图书/用户/借阅记录未找到 |
| 业务冲突 | 409 | 用户名已存在、图书已借出、数量超限 |
| 服务器错误 | 500 | 未预期的运行时错误 |

#### 4.1.4 前端错误处理

**Axios 响应拦截器：**

```js
// api/request.js
axios.interceptors.response.use(
  response => response.data,
  error => {
    if (error.response) {
      const { status, data } = error.response
      switch (status) {
        case 401:
          // Token 过期 → 清除登录态 → 跳转登录页
          localStorage.removeItem('token')
          localStorage.removeItem('userInfo')
          router.push('/login')
          ElMessage.error('登录已过期，请重新登录')
          break
        case 403:
          ElMessage.error('权限不足，无法执行此操作')
          break
        case 500:
          ElMessage.error('服务器繁忙，请稍后重试')
          break
        default:
          ElMessage.error(data.message || '操作失败')
      }
    } else if (error.code === 'ECONNABORTED') {
      ElMessage.error('请求超时，请检查网络连接')
    } else {
      ElMessage.error('网络连接异常，请检查网络')
    }
    return Promise.reject(error)
  }
)
```

**Vue 组件级错误边界：**

```js
// 使用 onErrorCaptured 钩子
import { onErrorCaptured, ref } from 'vue'

export function useErrorBoundary() {
  const error = ref(null)
  onErrorCaptured((err, instance, info) => {
    error.value = err
    console.error('[ErrorBoundary]', err, info)
    return false  // 阻止错误继续向上冒泡
  })
  return { error }
}
```

### 4.2 日志方案

#### 4.2.1 Winston 日志配置

```js
const winston = require('winston')
const path = require('path')

const logger = winston.createLogger({

    // 开发环境同时输出到控制台
    ...(process.env.NODE_ENV !== 'production' ? [
      new winston.transports.Console({
        format: winston.format.combine(

          winston.format.simple()

      })

  ]
```

#### 4.2.2 日志级别使用规范

| 级别 | 使用场景 |
|------|---------|
| error | 系统异常（数据库连接失败、未捕获异常），需要人工介入 |

#### 4.2.3 请求日志中间件

```js
function requestLogger(req, res, next) {
  const start = Date.now()
  res.on('finish', () => {
    const duration = Date.now() - start
    const logData = {
      method: req.method,
      url: req.originalUrl,
      status: res.statusCode,
      duration: `${duration}ms`,
      ip: req.ip,
      user: req.user?.id || 'anonymous'
    }
    if (res.statusCode >= 500) {
      logger.error('Request failed', logData)
    } else if (duration > 1000) {
      logger.warn('Slow request', logData)
    } else {
      logger.info('Request completed', logData)
    }
  })
  next()
}
```

## 5. 可扩展性设计

### 5.1 代码组织与模块化

#### 5.1.1 后端分层架构

```
server/src/
├── routes/       # 路由层：每个资源一个路由文件 (11个)
├── services/     # 业务逻辑层：每个领域一个 Service (9个)
├── models/       # 数据模型层：Sequelize Model 定义 (9个)
├── middleware/    # 中间件层：认证、授权、校验、日志、错误处理 (5个)
├── jobs/         # 定时任务层：逾期检测
├── utils/        # 工具函数：条码生成、JWT、响应封装
└── constants/    # 常量和枚举
```

依赖方向严格自上而下：`routes → services → models`，下层不感知上层。新增业务模块只需在对应层添加文件。

#### 5.1.2 前端模块化

```
src/
├── api/          # API 请求封装：每个资源一个文件 (10个)
├── views/        # 页面组件：按业务域分目录
├── components/   # 公共组件 (6个)
├── composables/  # 组合式函数：逻辑复用 (4个)
├── stores/       # Pinia 状态管理 (3个 store)
├── router/       # 路由配置：按角色分文件
└── utils/        # 工具函数
```

#### 5.1.3 扩展新功能的流程

以"添加通知管理模块"为例：

1. 后端：`models/Notification.js` → `services/notificationService.js` → `routes/notification.routes.js` → 在 `routes/index.js` 注册路由
2. 前端：`api/notifications.js` → `views/admin/notifications/` → `router/admin.routes.js` 添加路由
3. 前后端互不影响，新增 3 个文件 + 2 处修改即可

### 5.3 数据库扩展预留

| 扩展方向 | 预留措施 |

## 8. 技术风险与应对

| 风险 | 影响等级 | 发生概率 | 应对措施 |
|------|:-------:|:-------:|---------|
| SQLite 并发写入瓶颈 | 中 | 低 | WAL 模式；50 并发用户场景足够；监控响应时间；预留 PostgreSQL 切换路径 |
| SQLite 单点故障导致数据丢失 | 高 | 低 | 每日自动备份 + 手动备份 + 保留 7 天 |
| Node.js 单进程阻塞导致服务不可用 | 高 | 低 | systemd 自动重启；杜绝同步阻塞 I/O；监控 CPU/内存 |
| JWT 无状态无法主动撤销 | 中 | 低 | Token 有效期 24h；密码修改后旧 Token 自然过期；暴力破解锁定机制辅助 |
| 前端 XSS 攻击 | 中 | 低 | Vue 默认转义 + helmet 安全头 + CSP 策略 |
| 日志文件占满磁盘 | 中 | 中 | Winston 按大小切割（10MB）+ 最多保留 30 个文件（约 300MB） |
