## 1. 性能设计

### 1.1 性能目标

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 页面首屏加载时间 | < 2s（P95） | Lighthouse / Chrome DevTools Performance |
| API 响应时间（普通查询） | < 500ms（P95） | 服务端中间件计时 + 日志统计 |
| API 响应时间（复杂统计） | < 2s（P95） | 服务端中间件计时 |
| 并发借阅操作 | ≥ 50 并发 | k6 / ApacheBench 压测 |
| 批量导入（5000 条） | ≤ 30s | 服务端计时 |
| 数据库单条查询 | < 100ms（P99） | ORM 日志 / SQLite 慢查询分析 |

### 1.2 前端性能优化

#### 1.2.1 构建优化

| 策略 | 实现方式 | 预期收益 |
|------|---------|---------|
| 代码分割 | Vue Router 路由级懒加载 `() => import()` | 首屏 JS 体积减小 40-60% |
| Tree Shaking | Vite 默认 Rollup 打包，按需引入 Element Plus | 组件库体积减小 50%+ |
| 资源压缩 | Vite 构建启用 gzip/brotli 压缩，静态资源 CDN 化 | 传输体积减小 60-70% |
| 图片优化 | 封面图片使用 WebP 格式，缩略图限制尺寸（200×280） | 图片加载耗时 < 200ms |
| 字体图标 | Element Plus Icons 按需引入，不加载全量图标集 | 减少约 200KB 图标资源 |

#### 1.2.2 运行时优化

| 策略 | 实现方式 |
|------|---------|
| 虚拟滚动 | 长列表（图书检索结果、操作日志）使用 Element Plus `el-table` 的虚拟滚动或 `vue-virtual-scroller`，渲染可见行 + 缓冲区 |
| 防抖/节流 | 搜索输入框 300ms 防抖，滚动事件 100ms 节流 |
| 骨架屏 | 首屏加载时展示骨架屏，避免白屏等待 |
| 预加载 | 图书详情页封面图使用 `<link rel="preload">` 预加载 |
| Keep-Alive | 高频页面（图书列表、读者列表）使用 `<keep-alive>` 缓存组件状态，避免重复渲染 |
| 分页默认值 | 列表分页默认 20 条/页，避免单次加载过多数据 |

### 1.3 后端性能优化

#### 1.3.1 数据库查询优化

**索引设计**（SQLite WAL 模式下索引效率最高）：

| 表 | 索引字段 | 索引类型 | 命中场景 |
|-----|---------|---------|---------|
| books | isbn | UNIQUE | ISBN 精确查找、入库查重 |
| books | title | INDEX | 书名模糊搜索 |
| books | author | INDEX | 作者搜索 |
| books | category_code | INDEX | 分类浏览 |
| books | (status, category_code) | COMPOSITE | 分类 + 状态组合筛选 |
| book_copy | barcode | UNIQUE | 条码扫描查找（借还书核心操作） |
| book_copy | (book_id, status) | COMPOSITE | 按图书查可借副本数 |
| borrow | (copy_id, status) | COMPOSITE | 还书时查找有效借阅记录 |
| borrow | (reader_id, status) | COMPOSITE | 读者当前借阅查询、借书资格校验 |
| borrow | (status, due_date) | COMPOSITE | 逾期检测定时任务 |
| readers | reader_no | UNIQUE | 读者证号查找 |
| readers | phone | UNIQUE | 手机号查找 |
| audit_log | (user_id, created_at) | COMPOSITE | 操作日志按操作人+时间查询 |
| audit_log | created_at | INDEX | 操作日志按时间范围查询 |

**查询优化原则**：

- 所有 WHERE 条件字段必须有对应索引
- 模糊搜索使用 `LIKE 'keyword%'` 前缀匹配利用索引，避免 `LIKE '%keyword%'` 全表扫描
- 统计查询先 COUNT 再 JOIN，避免大表笛卡尔积
- borrow 和 audit_log 为高频写入/大表，定期清理归档（保留近 2 年数据）

#### 1.3.2 SQLite 并发优化

SQLite 默认只支持单写者。针对 50 并发借阅的需求，设计以下策略：

| 策略 | 配置/实现 |
|------|----------|
| WAL 模式 | `PRAGMA journal_mode=WAL;` 允许多读者 + 单写者并发，读不阻塞写 |
| 忙等待超时 | `PRAGMA busy_timeout=5000;` 写冲突时等待 5 秒而非立即报错 |
| 同步模式 | `PRAGMA synchronous=NORMAL;` 平衡性能与安全性（非 FULL） |
| 缓存大小 | `PRAGMA cache_size=-64000;` 分配 64MB 页面缓存 |
| 内存临时表 | `PRAGMA temp_store=MEMORY;` 临时表存储在内存中 |
| 外键约束 | `PRAGMA foreign_keys=ON;` 由应用层 Sequelize 事务保证 |
| 连接池 | Sequelize 连接池 `max=10, min=2`，借还等写操作串行化在事务中 |

**借阅并发控制**：

```
借书请求 1 ──┐
借书请求 2 ──┤
    ...      ├── Express 请求队列 ── SQLite 串行写入（WAL 模式下读不阻塞）
借书请求 N ──┘

写请求串行化：同一副本的借书操作通过数据库行级锁天然串行
读请求无阻塞：图书检索、读者查询等读请求在 WAL 模式下不受写操作影响
```

#### 1.3.3 缓存策略

| 缓存层级 | 缓存内容 | TTL | 实现方式 |
|---------|---------|-----|---------|
| 浏览器缓存 | 静态资源（JS/CSS/图片） | 1 年（带 hash 的文件名） | HTTP Cache-Control + ETag |
| 内存缓存 | 系统配置参数 | 5 分钟 | Node.js `Map` 对象，配置修改时主动失效 |
| 内存缓存 | 中图法分类树 | 永久（启动时加载，管理端修改时刷新） | 单例 Service 缓存 |
| API 响应缓存 | 统计概览数据（总藏书、总借出） | 1 分钟 | `Map` + 定时刷新 |
| API 响应缓存 | 借阅排行 | 5 分钟 | `Map` + 请求时检查过期 |
| 数据库查询缓存 | SQLite 页面缓存 | 自动管理 | `PRAGMA cache_size` 设定 |

**缓存更新策略**：

```
系统配置修改事件
  ├── 更新数据库 system_config 表
  ├── 清除内存配置缓存 → 下次读取时重新加载
  └── 日志记录配置变更

图书入库/借还事件
  ├── 事务提交成功
  └── 清除统计概览缓存 → 下次请求时重新计算
```

#### 1.3.4 Express 性能配置

| 配置项 | 设置 | 说明 |
|--------|------|------|
| 压缩中间件 | `compression` 中间件，gzip 级别 6 | 响应体压缩 |
| 请求体限制 | `express.json({ limit: '1mb' })` | 防止大请求体阻塞 |
| 超时设置 | `server.timeout = 30000` | 30 秒请求超时 |
| Keep-Alive | `server.keepAliveTimeout = 65000` | 复用 TCP 连接 |
| 静态资源 | `express.static` 设置 `maxAge: '1y'` | 静态资源强缓存 |

## 2. 安全性设计

### 2.1 认证与鉴权

#### 2.1.1 JWT Token 认证方案

```
┌─────────┐                              ┌──────────────┐
│  前端    │  1. POST /api/auth/login     │   后端        │
│ (Vue3)  │─────────────────────────────▶│  (Express)    │
│         │                              │              │
│         │  2. 返回 JWT Token            │  校验用户名   │
│         │  { token, user }             │  bcrypt 比对  │
│         │◀─────────────────────────────│  签发 JWT     │
│         │                              │              │
│         │  3. 后续请求携带 Token        │              │
│         │  Authorization: Bearer <jwt> │  JWT 中间件   │
│         │─────────────────────────────▶│  解析验证     │
│         │                              │  注入 req.user│
│         │  4. 返回数据或 401           │              │
│         │◀─────────────────────────────│              │
└─────────┘                              └──────────────┘
```

**Token 配置**：

| 参数 | 值 | 说明 |
|------|-----|------|
| 签名算法 | HS256 (HMAC-SHA256) | 对称签名，适合单体应用 |
| 密钥 | 32 字节随机字符串，环境变量 `JWT_SECRET` 注入 | 不硬编码、不入库 |
| 有效期（默认） | 24 小时 | `expiresIn: '24h'` |
| 有效期（记住我） | 7 天 | `expiresIn: '7d'` |
| Payload | `{ id, username, role }` | 最小化原则，不含敏感信息 |
| 存储位置 | 前端 `localStorage` | 配合 Axios 拦截器自动携带 |

**Token 生命周期**：

```
签发 → 有效期内正常使用 → 过期返回 401 → 前端跳转登录页
                                    → 支持 Token 刷新（可选）
登出 → 前端清除 localStorage（后端无状态，不存储 Token）
```

**安全增强措施**：

| 措施 | 实现 |
|------|------|
| 密码强度 | 注册时强制 ≥ 8 位，含字母和数字，前端 + 后端双重校验 |
| 密码存储 | bcrypt 加盐哈希，saltRounds = 10 |
| 登录失败锁定 | 同一账号连续失败 ≥ 5 次，锁定 15 分钟（`users.locked_until`） |
| 登录失败提示 | 统一返回"用户名或密码错误"，不区分用户名不存在还是密码错误 |
| 密码重置 | 仅管理员可操作，重置后强制用户下次登录修改密码 |
| JWT 吊销 | 管理员禁用用户后，下次 JWT 校验时拦截（查询 `users.status`） |

#### 2.1.2 RBAC 权限控制

| 角色 | 权限范围 |
|------|---------|
| 系统管理员 (admin) | 全部权限：用户管理、系统配置、日志审计、数据备份、所有业务操作 |
| 图书管理员 (librarian) | 图书入库/编目/下架、借书/还书、读者管理、逾期处理、统计查询 |
| 普通读者 (reader) | 图书检索、查看个人借阅/续借、个人信息维护、预约 |
| 游客 (未登录) | 仅图书检索浏览 |

**权限校验中间件链**：

```
请求进入
  ├── authMiddleware      → JWT 验证（游客可跳过）
  ├── roleMiddleware      → 角色权限校验（admin/librarian/reader）
  └── routeHandler        → 业务逻辑处理
```

**权限配置存储**：各角色的权限以常量定义在代码中（`config/permissions.js`），启动时加载。权限粒度精确到 API 端点级别。

### 2.2 输入校验

#### 2.2.1 双重校验策略

```
┌─────────────────────┐     ┌─────────────────────┐
│   前端校验（UX）      │     │   后端校验（安全）    │
│                     │     │                     │
│ Element Plus 表单   │     │ express-validator   │
│ 规则校验 + 提示      │     │ 中间件逐字段校验      │
│                     │     │                     │
│ 作用：提升用户体验    │     │ 作用：安全防线        │
│ 不可依赖前端校验      │     │ 必须全部通过后端校验   │
└─────────────────────┘     └─────────────────────┘
```

#### 2.2.2 后端校验规则

| 字段 | 校验规则 | 校验库 |
|------|---------|--------|
| 用户名 | `isLength({ min: 3, max: 50 })`, `matches(/^[a-zA-Z0-9_]+$/)` | express-validator |
| 密码 | `isLength({ min: 8, max: 32 })`, `matches(/[a-zA-Z]/)`, `matches(/[0-9]/)` | express-validator |
| 手机号 | `matches(/^1[3-9]\d{9}$/)` | express-validator |
| 邮箱 | `isEmail()`, `isLength({ max: 100 })` | express-validator |
| ISBN | `matches(/^(?:\d{10}|\d{13})$/)` | express-validator |
| 数字参数 | `isInt({ min: 1 })` | express-validator |
| 日期参数 | `isISO8601()` | express-validator |
| 枚举值 | `isIn(['active', 'frozen', 'lost', 'disabled'])` | express-validator |
| 分页参数 | `isInt({ min: 1 })` page, `isInt({ min: 1, max: 100 })` pageSize | express-validator |
| 文本字段 | `trim()`, `escape()` 防 XSS | express-validator/validator |

#### 2.2.3 特殊校验场景

| 场景 | 校验逻辑 |
|------|---------|
| 借阅数量上限 | 后端校验 `readers.current_borrowed + N <= readers.borrow_limit`，不可依赖前端传入值 |
| 操作权限 | 读者只能操作自己的续借/预约，后端通过 `req.user.id` 比对资源所有者 |
| 文件上传 | 封面图片限制 MIME 类型 `image/jpeg,image/png,image/webp`，大小 ≤ 5MB |
| 批量导入 | Excel/CSV 逐行校验，错误行记录并跳过，不中断整体导入 |

### 2.3 SQL 注入防护

**Sequelize ORM 内置防护**：所有数据库操作通过 Sequelize ORM 执行，使用参数化查询。

```javascript
// 安全做法：Sequelize 参数化查询
const book = await Book.findOne({
  where: { isbn: req.body.isbn }  // Sequelize 自动参数化，不可注入
});

// 危险做法（严禁）：原始 SQL 拼接
// const book = await sequelize.query(`SELECT * FROM books WHERE isbn = '${req.body.isbn}'`);
```

**规则**：
- 必须使用 Sequelize 的 Model API（`findOne`, `findAll`, `create`, `update`, `destroy`）而非原始 SQL
- 如确需原始查询，使用 `sequelize.query(sql, { replacements: {...} })` 参数绑定
- 统计查询使用 Sequelize 的聚合函数（`count`, `sum`, `max`）配合 `group` 选项
- 模糊搜索使用 Sequelize 的 `[Op.like]` 操作符

### 2.4 XSS 防护

| 防护层 | 措施 |
|--------|------|
| 前端输出 | Vue 3 默认对 `{{ }}` 插值进行 HTML 转义，v-html 仅用于可信内容 |
| 后端输入 | 所有文本字段入库前 `trim()` 处理，API 返回数据不做额外转义 |
| HTTP 头 | `helmet` 中间件设置 `X-XSS-Protection: 1; mode=block` |
| CSP | `helmet.contentSecurityPolicy()` 限制脚本来源，仅允许同源和 Element Plus CDN（如使用） |

**helmet 中间件配置**：

```javascript
const helmet = require('helmet');
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
      imgSrc: ["'self'", "data:", "blob:"],
    }
  }
}));
```

### 2.5 CSRF 防护

| 措施 | 说明 |
|------|------|
| SameSite Cookie | `SameSite=Strict` 防止跨站请求携带 Cookie |
| JWT in Header | Token 存储在 `Authorization` Header 中而非 Cookie，天然防 CSRF |
| CORS 配置 | 后端配置允许的 Origin 白名单（生产环境精确匹配，开发环境允许 localhost） |
| 关键操作确认 | 下架、删除、重置密码等操作前端二次确认弹窗 |

**CORS 配置**：

```javascript
const cors = require('cors');
app.use(cors({
  origin: process.env.NODE_ENV === 'production'
    ? ['https://library.example.com']
    : ['http://localhost:5173'],
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,
  maxAge: 86400
}));
```

### 2.6 其他安全措施

| 措施 | 实现 |
|------|------|
| 密码传输 | 登录/注册时密码在请求体中通过 HTTPS 传输，不在 URL 参数中传递 |
| 错误信息 | 生产环境 API 错误不暴露堆栈信息和内部细节，统一格式 `{ "code": 500, "message": "服务器内部错误" }` |
| 请求频率限制 | `express-rate-limit` 限制登录接口：同一 IP 每分钟最多 10 次 |
| 敏感信息 | `.env` 文件不提交 Git，`.gitignore` 忽略 `.env` 和 `*.db` 文件 |
| 依赖安全 | `npm audit` 定期检查依赖漏洞，CI 流程中加入 `npm audit --audit-level=high` |
| JWT 密钥 | 环境变量注入，长度 ≥ 32 字符，定期轮换（旧 Token 自然过期） |

**登录频率限制配置**：

```javascript
const rateLimit = require('express-rate-limit');
const loginLimiter = rateLimit({
  windowMs: 60 * 1000,    // 1 分钟窗口
  max: 10,                 // 最多 10 次
  message: { code: 429, message: '请求过于频繁，请稍后再试' }
});
app.use('/api/auth/login', loginLimiter);
```

### 3.2 错误处理

#### 3.2.1 统一错误响应格式

```json
{
  "code": 400,
  "message": "人类可读的错误描述",
  "data": null
}
```

#### 3.2.2 错误码体系

| HTTP 状态码 | code | 说明 | 场景 |
|------------|------|------|------|
| 200 | 200 | 成功 | 正常响应 |
| 201 | 201 | 创建成功 | POST 创建资源 |
| 400 | 400 | 请求参数错误 | 字段校验失败、格式错误 |
| 401 | 401 | 未认证 | Token 缺失/过期/无效 |
| 403 | 403 | 无权限 | 角色无权限访问该资源 |
| 404 | 404 | 资源不存在 | 图书/读者/记录不存在 |
| 409 | 409 | 资源冲突 | 用户名已存在、图书重复入库、并发冲突 |
| 422 | 422 | 业务规则限制 | 借阅超限、读者冻结、续借次数用完 |
| 429 | 429 | 请求过于频繁 | 登录频率限制 |
| 500 | 500 | 服务器内部错误 | 未预期的运行时错误 |
| 503 | 503 | 服务暂不可用 | 维护模式 |

#### 3.2.3 前端错误处理

| 场景 | 处理方式 |
|------|---------|
| API 网络错误 | Axios 响应拦截器统一 Toast 提示"网络连接异常，请检查网络" |
| 401 未认证 | 清除 Token，跳转登录页，提示"登录已过期，请重新登录" |
| 403 无权限 | 提示"您没有权限执行此操作"，不跳转 |
| 500 服务器错误 | 提示"服务器繁忙，请稍后重试"，不暴露错误详情 |
| 表单校验错误 | 后端返回字段级错误，前端映射到对应表单域高亮显示 |
| 页面崩溃 | Vue `onErrorCaptured` 全局捕获，展示错误边界组件 |

#### 3.2.4 后端错误处理

```javascript
// 全局错误处理中间件
app.use((err, req, res, next) => {
  // 记录完整错误到日志（含堆栈）
  logger.error({
    message: err.message,
    stack: err.stack,
    url: req.originalUrl,
    method: req.method,
    userId: req.user?.id
  });

  // 返回安全的错误给客户端
  const statusCode = err.statusCode || 500;
  const message = statusCode === 500
    ? '服务器内部错误'
    : err.message;

  res.status(statusCode).json({
    code: statusCode,
    message,
    data: null
  });
});
```

### 3.3 降级策略

| 场景 | 降级措施 |
|------|---------|
| ISBN 自动填充服务不可用 | 静默失败，允许手动填写所有字段 |
| 封面图片加载失败 | 展示默认占位图，不影响页面功能 |
| 统计查询超时 | 返回缓存数据（如 5 分钟前的结果），标记 `stale: true` |
| 操作日志写入失败 | 记录到 stderr，不阻塞主业务操作 |
| 数据库备份失败 | 日志告警，下次定时任务重试 |

### 3.4 容错机制

| 机制 | 实现 |
|------|------|
| 事务回滚 | 借书/还书操作使用 Sequelize 事务，任何步骤失败则完整回滚 |
| 数据库连接重试 | Sequelize `retry.max=3`，连接失败自动重试，间隔 2 秒 |
| 进程保活 | 使用 PM2 进程管理，`watch` 模式监听文件变化，崩溃自动重启 |
| 优雅关闭 | `process.on('SIGTERM')` 关闭 HTTP 服务 → 等待现有请求完成 → 关闭数据库连接 → 退出 |
| 未捕获异常 | `process.on('uncaughtException')` 记录日志后优雅关闭 |
| Promise 拒绝 | `process.on('unhandledRejection')` 记录日志 |

**PM2 配置（pm2.config.js）**：

```javascript
module.exports = {
  apps: [{
    name: 'library-api',
    script: './app.js',
    instances: 1,                    // 单实例（SQLite 单写限制）
    exec_mode: 'fork',
    watch: false,                    // 生产环境关闭 watch
    max_memory_restart: '512M',      // 内存超限自动重启
    env: { NODE_ENV: 'production' },
    error_file: './logs/pm2-error.log',
    out_file: './logs/pm2-out.log',
    log_date_format: 'YYYY-MM-DD HH:mm:ss'
  }]
};
```

| 备份频率 | 每日凌晨 2:00 自动备份 |

ls -t "$BACKUP_DIR"/library_*.db | tail -n +8 | xargs rm -f

### 5.2 日志规范

#### 5.2.1 日志框架

使用 Winston 结构化日志，支持不同级别：

```javascript
const winston = require('winston');
const logger = winston.createLogger({
  level: process.env.LOG_LEVEL || 'info',
  format: winston.format.combine(
    winston.format.timestamp(),
    winston.format.json()
  ),
  transports: [
    new winston.transports.File({ filename: 'logs/error.log', level: 'error' }),
    new winston.transports.File({ filename: 'logs/combined.log' }),
    new winston.transports.Console({ format: winston.format.simple() })
  ]
});
```

#### 5.2.2 日志级别与内容

| 级别 | 使用场景 | 示例 |
|------|---------|------|
| `error` | 系统错误，需立即关注 | 数据库连接失败、事务回滚、未捕获异常 |
| `warn` | 潜在问题，需关注 | 登录失败 ≥ 3 次、逾期超 30 天、磁盘空间 < 20% |
| `info` | 关键业务操作 | 借书/还书/续借、用户注册、图书入库、配置变更 |
| `debug` | 调试信息 | SQL 查询、请求参数、中间件处理时间（仅开发环境） |

#### 5.2.3 操作日志（audit_log 表）

与 Winston 应用日志互补，记录业务维度的操作审计：

| 操作类型 (action) | 记录内容 |
|-------------------|---------|
| `user.login` | 用户名、登录 IP、成功/失败 |
| `user.register` | 用户名、角色 |
| `book.import` | ISBN、书名、入库数量 |
| `book.withdraw` | 图书 ID、下架原因 |
| `borrow.create` | 读者 ID、副本 ID、借阅日期、应还日期 |
| `borrow.return` | 借阅记录 ID、归还日期、是否逾期 |
| `borrow.renew` | 借阅记录 ID、续借次数、新应还日期 |
| `config.update` | 配置键、旧值 → 新值 |
| `user.disable` / `user.enable` | 目标用户 ID |

**Winston 日志 vs audit_log 的分工**：

| 维度 | Winston 应用日志 | audit_log 表 |
|------|-----------------|-------------|
| 面向 | 开发/运维人员 | 系统管理员 |
| 目的 | 故障排查、性能分析 | 审计追溯、合规 |
| 存储 | 文件系统 | 数据库表 |
| 保留 | 滚动清理（30 天） | 永久保留（不可删除） |
| 查询 | 命令行 grep | 后台管理界面 |
