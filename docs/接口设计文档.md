## 1. API 概述

### 1.1 设计原则

| 原则 | 说明 |
|------|------|
| RESTful 风格 | 使用 HTTP 标准方法（GET/POST/PUT/PATCH/DELETE），资源使用名词复数形式 |
| JSON 数据格式 | 所有请求和响应使用 `Content-Type: application/json` |
| JWT 无状态认证 | 除注册和登录外，所有 API 在请求头中携带 `Authorization: Bearer <token>` |
| 统一响应结构 | 所有响应包裹在统一结构中，包含状态码、消息和数据体 |
| 版本化 URL | API 版本通过 URL 前缀区分，当前版本 `v1` |
| 资源嵌套 | 子资源通过路径嵌套表达归属关系，如 `/api/readers/:id/borrow-history` |
| HATEOAS 最小化 | 不在响应中嵌入链接，由前端基于角色路由控制可访问资源 |
| 幂等性 | GET/PUT/DELETE 操作保持幂等，重复调用不产生副作用 |

### 1.2 版本策略

| 项目 | 说明 |
|------|------|
| 当前版本 | v1 |
| 版本标识 | URL 路径前缀：`/api/v1` |
| 兼容策略 | 主版本号变更表示不兼容改动；次版本号变更表示向后兼容的新增 |
| 废弃策略 | 旧版本 API 在发布新版后保留 2 个大版本周期后下线 |

### 1.3 基础 URL

| 环境 | URL |
|------|-----|
| 开发环境 | `http://localhost:3000/api/v1` |
| 生产环境 | `https://<domain>/api/v1` |

> **简化约定**: 本文档后续接口路径省略 `/v1` 前缀（如 `/api/auth/login` 实为 `/api/v1/auth/login`），前端 Axios 实例的 `baseURL` 统一配置。

## 2. 统一响应格式

### 2.1 成功响应

```json
{
  "code": 200,
  "message": "ok",
  "data": { ... }
}
```

### 2.2 分页响应

```json
{
  "code": 200,
  "message": "ok",
  "data": {
    "list": [ ... ],
    "total": 150,
    "page": 1,
    "pageSize": 20
  }
}
```

### 2.3 错误响应

```json
{
  "code": 40401,
  "message": "未找到该图书",
  "data": null
}
```

### 2.4 HTTP 状态码映射

| HTTP 状态码 | 含义 | 使用场景 |
|-------------|------|---------|
| 200 | OK | GET/PUT/PATCH 成功 |
| 201 | Created | POST 创建成功 |
| 204 | No Content | DELETE 成功（无响应体） |
| 400 | Bad Request | 参数校验失败 |
| 401 | Unauthorized | 未认证或 Token 过期 |
| 403 | Forbidden | 权限不足 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 资源冲突（用户名重复等） |
| 422 | Unprocessable Entity | 业务规则校验失败 |
| 500 | Internal Server Error | 服务器内部错误 |

## 4. 认证方案

### 4.1 JWT 认证流程

```
客户端                              服务端
  │                                   │
  │  POST /api/auth/login             │
  │  { username, password }         → │
  │                                    │  验证凭据 → 生成 JWT
  │  ← { token, user_info }          │
  │                                    │
  │  GET /api/books                   │
  │  Authorization: Bearer <token>   → │
  │                                    │  解析 JWT → 鉴权 → 执行业务
  │  ← { books[] }                   │
```

### 4.2 Token 规范

| 属性 | 值 |
|------|-----|
| 算法 | HS256 |
| Payload | `{ userId, username, role, iat, exp }` |
| 短期 Token 有效期 | 24 小时 |
| "记住我" Token 有效期 | 7 天 |
| 传输方式 | HTTP Header `Authorization: Bearer <token>` |
| 存储位置 | 前端 `localStorage` |

### 4.3 RBAC 权限模型

| 角色 | 标识 | 权限范围 |
|------|------|---------|
| 系统管理员 | admin | 全部功能（含系统配置、用户管理、审计） |
| 图书管理员 | librarian | 图书管理、借阅管理、读者管理、统计查询 |
| 普通读者 | reader | 检索浏览、续借、借阅历史、个人信息、预约 |
| 游客 | guest | 仅限图书检索、浏览（无需登录） |

### 4.4 认证中间件

```
请求 → [JWT 解析] → [角色鉴权] → [控制器]
              │             │
              ▼             ▼
         无效/过期      角色不匹配
              │             │
              ▼             ▼
         401 Unauthorized  403 Forbidden
```

### 4.5 登录锁定策略

| 规则 | 值 |
|------|-----|
| 连续失败上限 | 5 次 |
| 锁定时长 | 15 分钟 |
| 锁定范围 | 按账号（非 IP） |
| 解锁 | 到期自动解锁 或 管理员手动解锁 |

## 5. 错误码定义

错误码采用 5 位数字编码，前 3 位对应 HTTP 状态码，后 2 位为业务细分码。

### 5.1 通用错误码 (10xx)

| 错误码 | HTTP 状态 | 消息 |
|--------|-----------|------|
| 40001 | 400 | 请求参数错误 |
| 40002 | 400 | 缺少必填参数 |
| 40003 | 400 | 参数格式不正确 |
| 40101 | 401 | 请先登录 |
| 40102 | 401 | 登录已过期，请重新登录 |
| 40103 | 401 | 认证失败 |
| 40301 | 403 | 权限不足，无法访问该资源 |
| 40302 | 403 | 该操作仅限管理员执行 |
| 40303 | 403 | 该操作仅限图书管理员执行 |
| 40401 | 404 | 请求的资源不存在 |
| 40901 | 409 | 资源已存在 |
| 42201 | 422 | 业务规则校验失败 |
| 50001 | 500 | 服务器内部错误 |
| 50002 | 500 | 数据库操作失败 |

### 5.2 认证错误码 (20xx)

| 错误码 | HTTP 状态 | 消息 |
|--------|-----------|------|

## 6. 接口详细定义

### 6.1 认证模块 `/api/auth`

---

#### 6.1.1 用户注册

```
POST /api/auth/register
```

**认证**: 否

**请求体**:

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| username | string | 是 | 用户名，唯一 |
| password | string | 是 | 密码，8-32 位含字母和数字 |
| name | string | 是 | 真实姓名 |
| phone | string | 是 | 手机号，1 开头 11 位 |
| email | string | 否 | 邮箱 |

**请求示例**:
```json
{
  "username": "zhangsan",
  "password": "Abc12345",
  "name": "张三",
  "phone": "13800138000",
  "email": "zhangsan@example.com"
}
```

**成功响应** (201):
```json
{
  "code": 201,
  "message": "注册成功",
  "data": {
    "userId": 1,
    "username": "zhangsan",
    "role": "reader"
  }
}
```

**错误响应**:
| 错误码 | 场景 |
|--------|------|
| 40920 | 用户名已被注册 |
| 40921 | 手机号已被注册 |
| 40020 | 密码强度不足 |
| 40001 | 参数格式错误（手机号、邮箱） |

---

#### 6.1.2 用户登录

```
POST /api/auth/login
```

**认证**: 否

**请求体**:

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| username | string | 是 | 用户名 |
| password | string | 是 | 密码 |
| rememberMe | boolean | 否 | 是否记住登录，默认 false |

**请求示例**:
```json
{
  "username": "zhangsan",
  "password": "Abc12345",
  "rememberMe": true
}
```

**成功响应** (200):
```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "expiresAt": "2026-07-05T12:00:00Z",
    "user": {
      "id": 1,
      "username": "zhangsan",
      "name": "张三",
      "role": "reader"
    }
  }
}
```

**错误响应**:
| 错误码 | 场景 |
|--------|------|
| 40120 | 用户名或密码错误 |
| 40121 | 账号已被禁用 |
| 40122 | 账号临时锁定（含剩余分钟数） |

---

#### 6.1.3 登出

```
POST /api/auth/logout
```

**认证**: 是

```javascript
// 请求拦截器：自动附加 Token
apiClient.interceptors.request.use(config => {
  const token = localStorage.getItem('token');
  if (token) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

// 响应拦截器：统一错误处理
apiClient.interceptors.response.use(
  response => response.data,
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('token');
      router.push('/login');
    }
    return Promise.reject(error.response?.data || error);
  }
);
```

### 8.2 时间格式

所有日期时间字段统一使用 ISO 8601 格式（UTC 时区）：`2026-07-04T09:00:00Z`。前端负责根据用户时区显示本地时间。

### 8.3 空值约定

- 列表为空时返回 `[]`（非 null）
- 对象为空时返回 `{}`（非 null）
- 不存在的数据字段统一返回 `null`
- 分页接口的 `total` 至少为 `0`

### 8.4 排序约定

- 默认按 `createdAt` 降序
- 支持的排序字段在各接口的 `sort` 参数中声明
- 不支持的排序字段将被忽略，不报错

### 8.5 并发安全

借书和还书操作涉及数据库事务，后端使用 SQLite 的 SERIALIZABLE 隔离级别确保数据一致性。前端在遇到 409/422 错误时应提示用户重试。
