## 1. 引言

### 1.1 目的

本文档旨在全面、系统地定义"图书管理系统"的功能需求与行为规范，为后续的架构设计、编码实现、测试验证和系统部署提供统一的需求基准。本文档的读者包括：

- **架构师**：依据本文档进行系统架构设计与技术选型验证
- **前端/后端开发者**：依据本文档理解功能边界和接口约定
- **测试工程师**：依据本文档编写测试用例和验收方案
- **项目管理人员**：依据本文档进行进度规划和里程碑管理

### 1.2 范围

本系统覆盖的核心业务领域：

| 领域 | 范围说明 |
|------|---------|
| 图书全生命周期管理 | 入库 → 编目 → 上架 → 下架 → 重新上架 |
| 借阅业务全流程管理 | 借出 → 归还 → 续借 → 逾期处理 → 罚款缴纳 |
| 读者信息管理 | 注册 → 信息维护 → 挂失 → 借阅历史查询 |
| 系统管理 | 用户账号管理 → 角色权限配置 → 系统参数配置 → 操作日志审计 |
| 多维查询与统计 | 图书检索 → 分类浏览 → 借阅排行 → 逾期统计 |

本系统**不包含**以下领域：

- 电子书在线阅读功能
- 图书采购/供应商管理
- 财务/费用结算系统（仅处理逾���罚款登记与缴纳标记）
- 移动端 App（仅提供 PC 端与平板设备的响应式 Web 界面）

### 1.5 整体描述

#### 1.5.1 产品视角

图书管理系统是一个基于 B/S 架构的单体 Web 应用，采用前后端分离架构。系统部署于单一服务器上，通过浏览器提供统一访问入口。后端提供 RESTful API，前端通过 HTTP/HTTPS 请求与后端交互，所有 API 请求通过 JWT Token 进行身份认证。

#### 1.5.2 功能综述

系统按模块划分为六大功能域：

| 功能域 | 核心功能 | 对应角色 |
|--------|---------|---------|
| 用户认证域 | 注册、登录、登出、Token 刷新 | 所有用户 |
| 图书管理域 | 入库、编目、下架、重新上架、查询 | 图书管理员 |
| 借阅管理域 | 借书、还书、续借、逾期检测、罚款管理 | 图书管理员、普通读者 |
| 读者管理域 | 注册、信息维护、挂失、借阅历史 | 系统管理员、图书管理员、普通读者 |
| 系统管理域 | 用户管理、角色权限、系统配置、操作日志、数据备份 | 系统管理员 |
| 查询统计域 | 多条件检索、借阅排行、逾期统计 | 所有角色 |

#### 1.5.3 用户特征

| 角色 | 技术水平 | 使用频率 | 使用场景 |
|------|---------|---------|---------|
| 系统管理员 | IT 背景，熟悉系统运维 | 低频（配置/审计时） | 桌面环境，管理后台 |
| 图书管理员 | 基础计算机操作能力 | 高频（日常操作） | 图书馆前台，扫码枪操作 |
| 普通读者 | 普通互联网用户 | 中频（借还时） | PC 或平板，自助检索与续借 |
| 游客 | 任意水平 | 低频（浏览时） | 任意设备，仅浏览检索 |

## 2. 详细需求

### 2.1 功能模块详细描述

#### 2.1.1 用户认证模块

##### 2.1.1.1 用户注册

| 属性 | 描述 |
|------|------|
| 触发者 | 游客 |
| 前置条件 | 用户未登录，注册页面可访问 |
| 输入 | 用户名(必填)、密码(必填，8-32 位含字母和数字)、姓名(必填)、手机号(必填)、邮箱(选填) |
| 处理逻辑 | 1. 校验用户名唯一性 2. 校验密码强度（至少 8 位，含字母和数字）3. 校验手机号格式（1 开头的 11 位数字）4. bcrypt 加密密码 5. 生成唯一读者证号（格式：R-YYYYMMDD-XXXXX）6. 创建用户记录，角色默认"普通读者"，状态"正常" 7. 记录操作日志 |
| 输出 | 注册成功提示，跳转登录页 |
| 异常 | 用户名已存在 → 提示"用户名已被注册"；手机号已存在 → 提示"手机号已被注册"；密码强度不足 → 提示规则 |
| 后置条件 | users 表新增一条记录，操作日志记录注册事件 |

##### 2.1.1.2 用户登录

| 属性 | 描述 |
|------|------|
| 触发者 | 所有已注册用户 |
| 前置条件 | 用户未登录，账号状态正常（非禁用） |
| 输入 | 用户名(必填)、密码(必填)、记住我(选填) |
| 处理逻辑 | 1. 根据用户名查询用户 2. bcrypt 比对密码 3. 校验账号状态（禁用/锁定）4. 生成 JWT Token（有效期 24h，"记住我"则 7 天）5. 记录登录日志（操作人、IP、时间）6. 失败计数逻辑：同一账号连续失败 ≥ 5 次，锁定 15 分钟 |
| 输出 | JWT Token + 用户信息（角色、姓名），跳转对应首页 |
| 异常 | 用户名不存在/密码错误 → "用户名或密码错误"（统一提示）；账号禁用 → "账号已被禁用"；账号锁定 → "账号已临时锁定，请 15 分钟后重试" |
| 后置条件 | Token 写入前端 localStorage；操作日志记录登录事件；失败计数器更新 |

##### 2.1.1.3 Token 校验与刷新

| 属性 | 描述 |
|------|------|
| 触发者 | 前端 Axios 拦截器（每次 API 请求） |
| 输入 | Authorization Header 中的 Bearer Token |
| 处理逻辑 | 1. 后端中间件解析 JWT 2. 校验签名和有效期 3. 从 Token 提取用户 ID 和角色 4. 注入 req.user 供后续中间件使用 |
| 输出 | req.user = { id, username, role } 或 401 |
| 异常 | Token 不存在 → 401, "请先登录"；Token 过期 → 401, "登录已过期，请重新登录"；Token 无效 → 401, "认证失败" |

#### 2.1.2 图书管理模块

##### 2.1.2.1 图书入库

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员 |
| 前置条件 | 管理员已登录 |
| 输入 | ISBN(必填)、书名(必填)、作者(必填)、出版社(必填)、出版日期(必填)、分类号(必填)、定价(必填)、页数(选填)、内容简介(选填)、入库数量(必填，默认1)、馆藏位置(选填)、封面图片(选····填) |
| 处理逻辑 | 1. 校验 ISBN 格式（10 位或 13 位）2. 查询 ISBN 是否已有记录 3. 若已有：提示是增加复本还是合并 4. 若新增：创建 book 记录 + 按入库数量逐册生成 book_copy 记录（含唯一图书条码）5. 每个副本初始状态：在库 6. 记录操作日志 |
| 输出 | 入库成功，返回生成的图书条码列表 |
| 异常 | ISBN 格式错误 → 提示；必填字段为空 → 高亮提示；入库数量 ≤ 0 → 提示 |

##### 2.1.2.2 图书编目

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员 |
| 前置条件 | 图书已入库 |
| 输入 | 图书 ID(必填)、丛书名(选填)、版次(选填)、语言(默认中文)、开本(选填)、主题词(选填)、附注(选填) |
| 处理逻辑 | 1. 根据分类号自动匹配分类名称（中图法分类表）2. 更新 book 表编目字段 3. 编目完成后，关联副本状态从"在库"更新为"可借" |
| 输出 | 编目成功确认 |
| 异常 | 图书不存在 → 提示"未找到该图书" |

##### 2.1.2.3 图书下架

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员 |
| 前置条件 | 图书存在且所有副本未被借出 |
| 输入 | 图书条码或图书 ID(必填)、下架原因(必选：破损/遗失/过时/其他)、备注(选填) |
| 处理逻辑 | 1. 校验所有副本状态均为"可借"或"在库" 2. 有副本借出中 → 阻止下架 3. 更新所有副本状态为"已下架" 4. 记录下架操作日志 |
| 输出 | 下架成功确认 |
| 异常 | 图书不存在 → 提示；有副本借出中 → "图书有副本正在借出，不可下架"；已下架 → "该图书已下架" |

#### 2.1.3 借阅管理模块

##### 2.1.3.1 借书

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员 |
| 前置条件 | 管理员已登录，读者账户正常 |
| 输入 | 读者证号/手机号(必填)、图书条码(必填，支持多本)、借阅天数(默认取系统配置) |
| 处理逻辑 | 1. 查找读者，校验状态（正常、无逾期、未达上限、无未缴罚款）2. 逐本校验：状态为"可借"、未被预约(或当前读者是预约者) 3. 计算应还日期 4. 开启数据库事务：创建 borrow 记录 → 更新 book_copy 状态为"借出" → 更新读者当前借阅数量 5. 提交事务 6. 记录操作日志 |
| 输出 | 借书成功，显示应还日期列表 |
| 异常 | 读者不存在/冻结/逾期/超限/欠费 → 对应提示；图书状态非可借 → "该图书无法借阅"；并发冲突（同一副本）→ 事务回滚，"该图书已被借出" |
| 后置条件 | borrow 表新增记录，book_copy 表状态更新，readers 表借阅计数 +N |

##### 2.1.3.2 还书

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员 |
| 前置条件 | 图书处于"借出"状态 |
| 输入 | 图书条码(必填) |
| 处理逻辑 | 1. 根据条码查找借阅中记录 2. 判断逾期：actual_return_date > due_date 3. 若逾期：计算逾期天数、罚款金额（0.1 元/天，上限为定价的 2 倍）4. 开启事务：更新 borrow 记录（状态→已归还，actual_return_date）→ 更新 book_copy 状态→ 可借 → 更新读者借阅计数 -1 → 若逾期则生成 fine 记录 5. 提交事务 6. 若逾期且自动冻结条件满足（≥ 30 天），冻结读者账户 |
| 输出 | 还书成功；若有逾期，显示罚款金额 |
| 异常 | 条码无效 → "未找到该图书"；无有效借阅记录 → 提示；已归还 → "该图书已归还" |

##### 2.1.3.3 续借

| 属性 | 描述 |
|------|------|
| 触发者 | 普通读者 |
| 前置条件 | 读者已登录，借阅记录有效 |
| 输入 | 借阅记录 ID |
| 处理逻辑 | 1. 校验记录状态为"借出中" 2. 校验未逾期 3. 校验续借次数 < 系统配置上限(默认 2 次) 4. 校验该图书未被其他读者预约 5. 新应还日期 = 原应还日期 + 续借天数(默认 30 天) 6. 续借次数 +1 7. 更新 borrow 记录 |
| 输出 | 续借成功，显示新应还日期 |
| 异常 | 逾期 → "逾期图书不可续借，请先归还"；次数用尽 → "续借次数已用完"；已被预约 → "该图书已被其他读者预约" |

#### 2.1.5 查询统计模块

##### 2.1.5.1 图书检索

| 属性 | 描述 |
|------|------|
| 触发者 | 所有角色（含游客） |
| 输入 | 关键词(选填)、作者(选填)、ISBN(选填)、分类(选填)、状态(选填)、出版年份范围(选填)、排序方式(选填)、分页参数 |
| 处理逻辑 | 1. 构建 WHERE 条件（模糊匹配书名/作者，精确匹配 ISBN，状态筛选，年份范围）2. LEFT JOIN 获取副本状态统计 3. 支持按书名、作者、出版日期排序 4. 分页返回（每页 20 条）5. 游客仅可查可借和借出状态 |
| 输出 | 分页图书列表 + 每本书的可借副本数 |

##### 2.1.5.2 借阅排行

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员、系统管理员 |
| 输入 | 时间范围(必填)、分类筛选(选填)、Top N(默认 20) |
| 处理逻辑 | 1. COUNT borrow 记录，GROUP BY book_id 2. 按借阅次数降序 3. LIMIT N 4. JOIN book 获取详情 |
| 输出 | 排行列表（排名、书名、作者、借阅次数） |

##### 2.1.5.3 逾期统计

| 属性 | 描述 |
|------|------|
| 触发者 | 图书管理员、系统管理员 |
| 输入 | 时间范围(选填)、统计维度（按读者/按图书） |
| 处理逻辑 | 1. 按维度 GROUP BY 2. 统计逾期次数、总罚款金额 3. 计算逾期率（逾期记录数 / 总借阅记录数） |
| 输出 | 统计报表（逾期总数、逾期率、读者/图书排行、罚款统计） |

## 3. 接口需求

### 3.1 API 设计原则

| 原则 | 说明 |
|------|------|
| RESTful | 使用 HTTP 标准方法（GET/POST/PUT/DELETE），资源名词复数形式 |

## 4. 数据需求

### 4.1 数据完整性约束

| 约束类型 | 说明 |
|---------|------|
| 实体完整性 | 每张表均有自增主键，确保实体唯一标识 |
| 参照完整性 | 外键关联确保引用一致：borrow.reader_id → readers.user_id，borrow.copy_id → book_copy.id，book_copy.book_id → books.id，fine.borrow_id → borrow.id |
| 域完整性 | 枚举值约束：状态字段限定取值范围；数值字段确保非负；日期字段格式一致 |
| 用户定义完整性 | ISBN 唯一、读者证号唯一、图书条码唯一、同一借阅记录最多一次续借、借阅数量不超过上限 |

### 4.2 数据安全

| 维度 | 措施 |
|------|------|
| 密码存储 | bcrypt 加盐哈希，不可逆，不可还原 |
| 敏感数据 | Token 不在服务端存储；日志不记录密码明文 |
| 数据隔离 | 读者只能访问自己的借阅记录和个人信息 |
| 审计追溯 | 所有关键操作记录到 audit_log，不可删除 |
| 数据备份 | 每日自动备份 SQLite 文件，保留最近 7 天 |

### 4.3 数据容量预估

| 实体 | 规模 | 存储预估 |
|------|------|---------|
| users | 5 万 | ~10 MB |
| books | 10 万 | ~50 MB |
| book_copy | 15 万（含复本） | ~15 MB |
| borrow | 100 万 | ~80 MB |
| fine | 10 万 | ~5 MB |
| audit_log | 200 万 | ~100 MB |
| reserve | 5 万 | ~3 MB |
| **合计** | — | **~263 MB** |

SQLite 单库支持最大 140 TB，完全满足本系统的数据容量需求。

### 4.4 事务要求

| 操作 | 事务范围 | 隔离级别 |
|------|---------|---------|
| 借书 | borrow 写入 + book_copy 状态更新 + readers 计数更新 | SERIALIZABLE |
| 还书 | borrow 状态更新 + book_copy 状态更新 + readers 计数更新 + fine 写入（如逾期） | SERIALIZABLE |
| 续借 | borrow 记录更新 | SERIALIZABLE |
| 批量入库 | books 写入 + book_copy 批量写入 | SERIALIZABLE |
| 系统配置更新 | system_config 批量更新 | SERIALIZABLE |

所有涉及多表写入的操作必须包含在一个数据库事务中，确保原子性：要么全部成功，要么全部回滚。

## 5. 约束与假设

### 5.1 设计约束

1. **前后端分离**：前端与后端通过 HTTP REST API 通信，通信数据格式为 JSON
2. **JWT 认证**：所有需要认证的接口必须在请求头中携带有效的 JWT Token
3. **数据库**：使用 SQLite 3，数据文件存储在服务器本地磁盘
4. **单机部署**：系统设计为单体应用，不涉及分布式架构
5. **操作日志不可删除**：审计日志一经记录则永久保留

### 5.2 业务假设

1. 借阅操作由图书管理员执行，读者不直接操作借书（需到馆借阅）
2. 续借操作可由读者在线自助完成
3. 逾期检测由系统定时任务自动执行（建议每日 00:00 执行）
4. 读者类型与借阅上限的关系为系统预置，管理员可手动调整
5. 罚款仅为登记与缴纳标记，实际支付不通过本系统处理
6. 系统不记录用户密码的明文，不可找回密码（仅管理员可重置）

### 5.3 外部接口依赖

| 依赖项 | 说明 | 影响 |
|--------|------|------|
| ISBN 查询接口 | 入库时根据 ISBN 自动填充图书信息 | 可选，失败不影响手动录入 |
