## 1. 数据库概要

### 1.1 选型与配置

| 项目 | 说明 |
|------|------|
| 数据库 | SQLite 3 (≥ 3.35) |
| 连接模式 | WAL (Write-Ahead Logging) |
| 字符编码 | UTF-8 |
| 文件位置 | `server/data/library.db` |
| 备份策略 | 每日全量备份至 `server/backup/`，保留 7 天 |
| 超时设置 | busy_timeout = 5000ms |
| 缓存大小 | cache_size = -8000 (~8MB) |
| 同步模式 | synchronous = NORMAL (WAL 模式安全) |
| 外键 | PRAGMA foreign_keys = ON |

### 1.2 设计原则

- **第三范式**：消除数据冗余，保证数据一致性；适度冗余 `books.total_copies`/`available_copies` 和 `readers.current_borrowed` 以提升查询性能
- **外键约束**：开启 SQLite 外键约束，所有跨表引用通过外键维护参照完整性
- **级联删除**：`book_copy` → `book_id` 级联删除；`fine` → `borrow_id` 级联删除；`readers` → `user_id` 级联删除
- **软删除**：`users` 表使用 `deleted_at` 字段替代物理删除，保留历史关联
- **审计不可删**：`audit_log` 表不暴露 DELETE 接口，日志记录永久保留
- **枚举约束**：所有枚举字段使用 CHECK 约束限制取值范围
- **数值约束**：借阅数量、续借次数、罚款金额等业务数值通过 CHECK 约束保证合法

## 3. 表结构详细设计

### 3.1 用户表 (users)

#### 3.1.1 字段定义

| 序号 | 字段名 | 类型 | 长度 | 允许空 | 默认值 | PK | FK | UQ | 索引 | 自增 | 说明 |
|------|--------|------|------|--------|--------|----|----|----|------|------|------|
| 1 | id | INTEGER | — | 否 | — | ● | — | — | — | ● | 用户唯一 ID |
| 2 | username | VARCHAR | 50 | 否 | — | — | — | ● | ● | — | 登录用户名，3-50 字符 |
| 3 | password | VARCHAR | 255 | 否 | — | — | — | — | — | — | bcrypt 哈希，salt 10 轮，固定 60 字符 |
| 4 | role | VARCHAR | 20 | 否 | 'reader' | — | — | — | ● | — | admin / librarian / reader |
| 5 | status | VARCHAR | 20 | 否 | 'active' | — | — | — | ● | — | active / disabled / locked |
| 6 | login_attempts | INTEGER | — | 否 | 0 | — | — | — | — | — | 连续登录失败次数，成功登录后重置 |
| 7 | locked_until | DATETIME | — | 是 | NULL | — | — | — | — | — | 锁定到期时间，NULL 表示未锁定 |
| 8 | created_at | DATETIME | — | 否 | CURRENT_TIMESTAMP | — | — | — | — | — | 创建时间，建表时由 DEFAULT 设置 |
| 9 | updated_at | DATETIME | — | 否 | CURRENT_TIMESTAMP | — | — | — | — | — | 更新时间，更新时由应用层设置 |
| 10 | deleted_at | DATETIME | — | 是 | NULL | — | — | — | — | — | 软删除标记，非 NULL 表示已删除 |

#### 3.1.2 DDL

```sql
CREATE TABLE IF NOT EXISTS users (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    username        VARCHAR(50)  NOT NULL UNIQUE,
    password        VARCHAR(255) NOT NULL,
    role            VARCHAR(20)  NOT NULL DEFAULT 'reader',
    status          VARCHAR(20)  NOT NULL DEFAULT 'active',
    login_attempts  INTEGER      NOT NULL DEFAULT 0,
    locked_until    DATETIME              DEFAULT NULL,
    created_at      DATETIME     NOT NULL DEFAULT (datetime('now')),
    updated_at      DATETIME     NOT NULL DEFAULT (datetime('now')),
    deleted_at      DATETIME              DEFAULT NULL,
    CHECK (role IN ('admin', 'librarian', 'reader')),
    CHECK (status IN ('active', 'disabled', 'locked')),
    CHECK (length(username) >= 3 AND length(username) <= 50)
);
```

#### 3.1.3 约束说明

| 约束 | 类型 | 说明 |
|------|------|------|
| username UNIQUE | 唯一索引 | 保证登录名全局唯一 |
| role CHECK | 检查约束 | 限定角色为三种预置角色 |
| status CHECK | 检查约束 | 限定三种账户状态 |
| username 长度 | 应用层校验 | ≥ 3 且 ≤ 50，CREATE 时由 express-validator 校验 |
| password 长度 | 应用层校验 | ≥ 8 且 ≤ 32，必须含字母和数字 |

#### 3.1.4 索引

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| idx_users_username | username | UNIQUE | 登录认证：`WHERE username = ?` |
| idx_users_role | role | NORMAL | 用户列表筛选：`WHERE role = ?` |
| idx_users_status | status | NORMAL | 活跃用户筛选：`WHERE status = ?` |
| idx_users_deleted_at | deleted_at | NORMAL | 软删除过滤：`WHERE deleted_at IS NULL` |

#### 3.1.5 业务规则

- 软删除：`DELETE` 操作实际执行 `UPDATE users SET deleted_at = datetime('now')`，查询时默认过滤 `WHERE deleted_at IS NULL`
- 登录失败：`login_attempts` 累计到 5 次后，`status` 改为 `locked`，`locked_until` 设为当前时间 + 15 分钟
- 锁定到期：登录时检查 `locked_until`，若已过期且 `status = 'locked'`，自动重置为 `active` 并清零 `login_attempts`
- 密码存储：使用 bcryptjs 加密，hash 结果固定 60 字符，无需存储盐值

#### 3.2.1 字段定义

| 序号 | 字段名 | 类型 | 长度 | 允许空 | 默认值 | PK | FK | UQ | 索引 | 自增 | 说明 |
|------|--------|------|------|--------|--------|----|----|----|------|------|------|
| 1 | user_id | INTEGER | — | 否 | — | ● | ● | — | — | — | 关联 users.id，CASCADE DELETE |
| 2 | reader_no | VARCHAR | 25 | 否 | — | — | — | ● | ● | — | 读者证号，格式 R-YYYYMMDD-XXXXX |
| 3 | name | VARCHAR | 50 | 否 | — | — | — | — | — | — | 读者真实姓名 |
| 4 | phone | VARCHAR | 20 | 否 | — | — | — | ● | ● | — | 手机号，1 开头 11 位数字 |
| 5 | email | VARCHAR | 100 | 是 | NULL | — | — | — | — | — | 电子邮箱 |
| 6 | reader_type | VARCHAR | 20 | 否 | 'student' | — | — | — | — | — | student / teacher / staff / external |
| 7 | borrow_limit | INTEGER | — | 否 | 5 | — | — | — | — | — | 最大可借册数 |
| 8 | current_borrowed | INTEGER | — | 否 | 0 | — | — | — | — | — | 当前在借数，CHECK ≥ 0 |
| 9 | status | VARCHAR | 20 | 否 | 'active' | — | — | — | ● | — | active / frozen / lost / disabled |
| 10 | created_at | DATETIME | — | 否 | CURRENT_TIMESTAMP | — | — | — | — | — | 注册时间 |
| 11 | updated_at | DATETIME | — | 否 | CURRENT_TIMESTAMP | — | — | — | — | — | 更新时间 |

#### 3.2.2 DDL

```sql
CREATE TABLE IF NOT EXISTS readers (
    user_id          INTEGER      PRIMARY KEY,
    reader_no        VARCHAR(25)  NOT NULL UNIQUE,
    name             VARCHAR(50)  NOT NULL,
    phone            VARCHAR(20)  NOT NULL UNIQUE,
    email            VARCHAR(100)          DEFAULT NULL,
    reader_type      VARCHAR(20)  NOT NULL DEFAULT 'student',
    borrow_limit     INTEGER      NOT NULL DEFAULT 5,
    current_borrowed INTEGER      NOT NULL DEFAULT 0,
    status           VARCHAR(20)  NOT NULL DEFAULT 'active',
    created_at       DATETIME     NOT NULL DEFAULT (datetime('now')),
    updated_at       DATETIME     NOT NULL DEFAULT (datetime('now')),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CHECK (reader_type IN ('student', 'teacher', 'staff', 'external')),
    CHECK (current_borrowed >= 0 AND current_borrowed <= borrow_limit),
    CHECK (status IN ('active', 'frozen', 'lost', 'disabled'))
);
```

#### 3.2.3 约束说明

| 约束 | 类型 | 说明 |
|------|------|------|
| user_id PRIMARY KEY + FK | 主键+外键 | 与 users.id 一对一，ON DELETE CASCADE |
| reader_no UNIQUE | 唯一约束 | 读者证号全局唯一 |
| phone UNIQUE | 唯一约束 | 手机号全局唯一 |
| reader_type CHECK | 检查约束 | 限定四种读者类型 |
| current_borrowed CHECK | 检查约束 | ≥ 0 且 ≤ borrow_limit |
| borrow_limit | 默认值 | 按 reader_type 在应用层设置：student=5, teacher=10, staff=5, external=3 |

#### 3.2.4 索引

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| idx_readers_reader_no | reader_no | UNIQUE | 读者证号查询、借书时扫描证号 |
| idx_readers_phone | phone | UNIQUE | 手机号查询、借书时手机号识别 |
| idx_readers_status | status | NORMAL | 按状态筛选：`WHERE status = ?` |
| idx_readers_name | name | NORMAL | 姓名模糊搜索：`WHERE name LIKE ?` |

#### 3.2.5 读者证号生成规则

```
格式：R-YYYYMMDD-XXXXX
其中：
  R         — 固定前缀，Reader 首字母
  YYYYMMDD  — 注册日期
  XXXXX     — 当日 5 位自增序号（从 00001 开始）

示例：R-20260704-00001（2026年7月4日注册的第 1 位读者）

生成 SQL：
  SELECT 'R-' || strftime('%Y%m%d', 'now') || '-' ||
    printf('%05d', COALESCE(
      (SELECT CAST(SUBSTR(reader_no, -5) AS INTEGER) + 1
       FROM readers WHERE reader_no LIKE 'R-' || strftime('%Y%m%d', 'now') || '-%'
       ORDER BY reader_no DESC LIMIT 1),
      1
    ));
```

#### 3.2.6 业务规则

- 读者注册：`POST /api/auth/register` 自动创建 users + readers 记录，role 固定 'reader'
- 管理员创建：`POST /api/users` 当 role='reader' 时同时创建 readers 记录
- borrow_limit 按 reader_type 自动赋值：student=5, teacher=10, staff=5, external=3
- current_borrowed 由借书/还书操作维护，禁止应用层直接修改
- 逾期冻结：`current_borrowed` 中有 overdue 记录且逾期 ≥ 30 天，status 自动标记为 'frozen'
- 挂失：管理员调用 `PUT /api/readers/:id/status` 设置 status='lost'；一旦挂失，禁止借阅
- 禁用：当 users.status = 'disabled' 时，对应 readers.status 也应为 'disabled'（应用层同步）

### 3.3 图书分类表 (categories)

#### 3.3.1 字段定义

| 序号 | 字段名 | 类型 | 长度 | 允许空 | 默认值 | PK | FK | UQ | 索引 | 自增 | 说明 |

| 6 | sort_order | INTEGER | — | 否 | 0 | — | — | — | — | — | 同级排序权重 |

#### 3.3.2 DDL

```sql
CREATE TABLE IF NOT EXISTS categories (
    id          INTEGER      PRIMARY KEY AUTOINCREMENT,
```

#### 3.3.3 约束说明

| 约束 | 类型 | 说明 |
|------|------|------|
| code UNIQUE | 唯一约束 | 分类号全局唯一 |
| parent_id FK | 自引用外键 | 构成树形结构，ON DELETE SET NULL |
| level CHECK | 检查约束 | 层级 ≥ 1 |

#### 3.3.4 索引

| 索引名 | 字段 | 类型 | 用途 |
|--------|------|------|------|
| idx_categories_code | code | UNIQUE | 按分类号精确查询 |
| idx_categories_parent_id | parent_id | NORMAL | 查询子分类：`WHERE parent_id = ?` |
| idx_categories_level | level | NORMAL | 按层级筛选：`WHERE level = ?` |

#### 3.3.5 树形查询

```
查询一级分类：
  SELECT * FROM categories WHERE parent_id IS NULL ORDER BY sort_order

查询指定分类的子分类：
  SELECT * FROM categories WHERE parent_id = ? ORDER BY sort_order

获取完整树（应用层构建，或使用递归 CTE）：
  WITH RECURSIVE tree AS (
    SELECT *, 0 AS depth FROM categories WHERE parent_id IS NULL
    UNION ALL
    SELECT c.*, t.depth + 1 FROM categories c JOIN tree t ON c.parent_id = t.id
  ) SELECT * FROM tree ORDER BY depth, sort_order;
```

#### 3.3.6 与 books 表关系

#### 3.4.5 冗余字段维护策略

`total_copies` 和 `available_copies` 是性能优化的冗余字段。维护方式：

| 操作 | total_copies 变化 | available_copies 变化 |
|------|-------------------|----------------------|
| 图书入库（创建 book_copy） | +N | +N（若副本状态为 stock/available） |
| 增加副本（API 增加） | +N | +N |
| 编目完成（copy status stock→available） | 不变 | +1 |
| 借书（copy status available→borrowed） | 不变 | -1 |
| 还书（copy status borrowed→available） | 不变 | +1 |
| 副本下架（copy status →withdrawn） | 不变 | -1（若原为 available） |
| 重新上架（copy status withdrawn→available） | 不变 | +1 |
| 图书下架（全部副本 withdrawn） | 不变 | 0 |

**维护方式**：由 Service 层在事务中手动 UPDATE，不要使用数据库触发器（Sequelize 事务中触发器不可靠）。示例：

```javascript
// BorrowService.borrowBooks() 事务内
await Book.increment({ available_copies: -n }, { where: { id: bookId }, transaction });
```

#### 3.5.5 副本状态流转

```
                         ┌──────────┐
                         │   stock   │  (入库初始状态，尚未编目)
                         └─────┬─────┘
                               │ 编目完成
                               ▼
               ┌───────────────────────┐
               │      available        │  (可借状态)
               └───┬──────────────┬────┘
                   │              │
         借书      │              │ 下架
                   ▼              ▼
               ┌──────────┐  ┌───────────┐
               │ borrowed │  │ withdrawn │  (已下架)
               └────┬─────┘  └─────┬─────┘
                    │              │
          还书      │     重新上架  │
                    ▼              │
               ┌──────────┐        │
               │ available│◀───────┘
               └──────────┘

流转规则:
  stock → available    : 图书编目完成后自动转换
  available → borrowed : 借书操作触发
  borrowed → available : 还书操作触发
  available → withdrawn: 下架操作触发（仅可借/在库状态）
  stock → withdrawn    : 未编目图书也可直接下架
  withdrawn → available: 重新上架操作触发
  borrowed → withdrawn : 禁止（有副本借出时不可下架）
```

#### 3.5.6 条码生成规则

```
格式：BC-YYYYMMDD-XXXXX
其中：
  BC        — 固定前缀，BarCode 缩写
  YYYYMMDD  — 入库日期
  XXXXX     — 当日 5 位自增序号（从 00001 开始）

示例：BC-20260704-00001

生成 SQL：
  SELECT 'BC-' || strftime('%Y%m%d', 'now') || '-' ||
    printf('%05d', COALESCE(
      (SELECT CAST(SUBSTR(barcode, -5) AS INTEGER) + 1
       FROM book_copy WHERE barcode LIKE 'BC-' || strftime('%Y%m%d', 'now') || '-%'
       ORDER BY barcode DESC LIMIT 1),
      1
    ));
```

```
                    ┌───────────┐

                          │
          ┌───────────────┼───────────────┐

    └──────────┘   └────┬─────┘          │

```

**逾期标记时机建议**：
- 方案 A（推荐）：定时任务每日 00:00 执行 `UPDATE borrow SET status='overdue' WHERE status='borrowing' AND due_date < date('now')`
- 方案 B：还书/续借时动态判断，不实际更新 status，由 `CASE WHEN` 在查询时计算

建议采用**方案 B**为主，方案 A 为辅（用于统计查询优化）。

#### 3.7.5 罚款计算规则

```
罚款金额 = MIN(逾期天数 × 逾期费率, 图书定价 × 罚款上限倍数)

其中：
  逾期费率        = system_config.overdue_fine_rate（默认 0.1 元/天）
  罚款上限倍数    = system_config.fine_max_multiple（默认 2.0）
  图书定价        = books.price

示例：
  图书定价 129 元，逾期 5 天
  罚款金额 = MIN(5 × 0.1, 129 × 2.0) = MIN(0.5, 258) = 0.5 元

  图书定价 20 元，逾期 100 天
  罚款金额 = MIN(100 × 0.1, 20 × 2.0) = MIN(10, 40) = 10 元
```

#### 3.7.6 业务规则

- 还书时自动计算：`return_date > due_date` 时自动生成 fine 记录
- 逾期 ≥ 30 天自动冻结读者账户（`readers.status = 'frozen'`）
- 罚款缴纳：`PUT /api/fines/:id/pay` 设置 `status='paid'`, `paid_at=now()`
- 实际支付不通过本系统处理，接口仅做登记
- `reader_id` 冗余字段由应用层在创建 fine 时自动填充

```sql
CREATE TABLE IF NOT EXISTS reserve (
    id            INTEGER     PRIMARY KEY AUTOINCREMENT,
    reader_id     INTEGER     NOT NULL,
    book_id       INTEGER     NOT NULL,
    copy_id       INTEGER              DEFAULT NULL,
    reserve_date  DATETIME    NOT NULL DEFAULT (datetime('now')),
    expire_date   DATE        NOT NULL,
    status        VARCHAR(20) NOT NULL DEFAULT 'waiting',
    created_at    DATETIME    NOT NULL DEFAULT (datetime('now')),

    FOREIGN KEY (book_id)   REFERENCES books(id),
    FOREIGN KEY (copy_id)   REFERENCES book_copy(id),
```

#### 3.8.5 预约状态流转

```
                ┌─────────┐
     create()──▶│ waiting │
                └────┬────┘
                     │
         ┌───────────┼───────────┐
         │           │           │
    cancel()    fulfill()    expire()
         │           │           │
         ▼           ▼           ▼
    ┌─────────┐ ┌─────────┐ ┌─────────┐
    │cancelled│ │fulfilled│ │ expired │
    └─────────┘ └─────────┘ └─────────┘

触发条件:
  create()   → POST /api/reserves，读者主动预约
  cancel()   → DELETE /api/reserves/:id，读者/管理员取消
  fulfill()  → 借书时若当前读者是预约者，自动 fulfill
  expire()   → 定时任务 / 查询时判断：expire_date < now() 且 status='waiting'
```

#### 3.8.6 预约规则

- 预约到期日 = 当前日期 + `system_config.reserve_expire_days`（默认 3 天）
- 同一读者对同一本书只能有一个 waiting 状态预约
- 图书馆当前有可借副本时，不允许预约（直接借阅即可）
- 预约到期后状态变为 `expired`（定时任务或查询时触发）
- 借书时检查：若当前借书人是该书的预约者，自动将预约状态改为 `fulfilled`

#### 3.10.4 配置使用方式

- 配置值统一以 TEXT 存储
- 应用层启动时加载全部配置到内存 Map
- 每次配置更新后刷新内存缓存
- 配置读取优先从内存缓存获取，缓存未命中则查库
- Sequelize 服务层实现示例：

```javascript
// system.service.js
let configCache = new Map();

async function loadConfigs() {
  const configs = await SystemConfig.findAll();
  configs.forEach(c => configCache.set(c.config_key, c.config_value));
}

function getConfig(key, defaultValue = null) {
  return configCache.has(key) ? configCache.get(key) : defaultValue;
}

function getConfigInt(key, defaultValue = 0) {
  return parseInt(getConfig(key, String(defaultValue)), 10);
}

function getConfigFloat(key, defaultValue = 0) {
  return parseFloat(getConfig(key, String(defaultValue)));
}

async function updateConfigs(configs, userId, transaction) {
  for (const { key, value } of configs) {
    await SystemConfig.update(
      { config_value: String(value), updated_by: userId, updated_at: new Date() },
      { where: { config_key: key }, transaction }
    );
    configCache.set(key, String(value));
  }
}
```

## 5. 视图设计

### 5.1 借阅详情视图 (v_borrow_detail)

用于借阅记录列表和报表导出，JOIN 所有关联表。

```sql
CREATE VIEW IF NOT EXISTS v_borrow_detail AS
SELECT
    b.id            AS borrow_id,
    r.user_id       AS reader_user_id,
    r.reader_no,
    r.name          AS reader_name,
    r.phone         AS reader_phone,
    bk.id           AS book_id,
    bk.title        AS book_title,
    bk.author       AS book_author,
    bk.isbn,
    bk.publisher,
    bk.price,
    bc.id           AS copy_id,
    bc.barcode,
    bc.location,
    u.username      AS operator_name,
    b.borrow_date,
    b.due_date,
    b.return_date,
    b.status        AS borrow_status,
    b.renew_count,
    CASE WHEN b.status IN ('borrowing','overdue') AND b.due_date < date('now')
         THEN CAST(julianday('now') - julianday(b.due_date) AS INTEGER)
         ELSE 0
    END AS overdue_days,
    f.id            AS fine_id,
    f.amount        AS fine_amount,
    f.status        AS fine_status,
    b.created_at,
    b.updated_at
FROM borrow b
JOIN readers r     ON b.reader_id = r.user_id
JOIN book_copy bc  ON b.copy_id = bc.id
JOIN books bk      ON bc.book_id = bk.id
JOIN users u       ON b.operator_id = u.id
LEFT JOIN fine f   ON b.id = f.borrow_id;
```

### 5.2 馆藏统计视图 (v_inventory_stats)

用于概览面板和库存报表。

```sql
CREATE VIEW IF NOT EXISTS v_inventory_stats AS
SELECT
```

### 5.3 读者借阅概览视图 (v_reader_overview)

用于读者管理和概览面板。

```sql
CREATE VIEW IF NOT EXISTS v_reader_overview AS
SELECT
    r.user_id,
    r.reader_no,
    r.name,
    r.phone,
    r.reader_type,
    r.borrow_limit,
    r.current_borrowed,
    r.status                 AS reader_status,
    COUNT(CASE WHEN b.status = 'borrowing' THEN 1 END)    AS active_borrows,
    COUNT(CASE WHEN b.status = 'overdue'   THEN 1 END)    AS overdue_borrows,
    COUNT(CASE WHEN f.status  = 'unpaid'   THEN 1 END)    AS unpaid_fines,

    u.created_at,
    r.created_at             AS registered_at
FROM readers r
JOIN users u          ON r.user_id = u.id
LEFT JOIN borrow b   ON r.user_id = b.reader_id
LEFT JOIN fine f     ON r.user_id = f.reader_id
WHERE u.deleted_at IS NULL
GROUP BY r.user_id;
```

### 7.1 事务操作清单

| 业务操作 | 涉及表 | 操作类型 | 事务级别 | 回滚条件 |
|---------|--------|---------|---------|---------|
| 用户注册 | users, readers, audit_log | INSERT×3 | SERIALIZABLE | 任何一步失败 |
| 图书入库 | books, book_copy(n) | INSERT×1 + BATCH INSERT | SERIALIZABLE | ISBN 冲突 / 条码生成冲突 |
| 图书编目 | books, book_copy(n) | UPDATE×1 + BATCH UPDATE | SERIALIZABLE | 并发编目冲突 |
| 图书下架 | books, book_copy(n), audit_log | UPDATE×2 + INSERT×1 | SERIALIZABLE | 有副本借出中 |
| 借书 | borrow(n), book_copy(n), readers, books, audit_log | INSERT×n + UPDATE×3 + INSERT×1 | SERIALIZABLE | 任意副本校验失败 |
| 还书 | borrow, book_copy, readers, books, fine(条件), audit_log | UPDATE×4 + INSERT×1(条件) + INSERT×1 | SERIALIZABLE | 并发还书冲突 |
| 续借 | borrow, audit_log | UPDATE×1 + INSERT×1 | SERIALIZABLE | 并发续借冲突 |
| 罚款缴纳 | fine, audit_log | UPDATE×1 + INSERT×1 | SERIALIZABLE | 重复缴纳 |
| 创建预约 | reserve, audit_log | INSERT×2 | SERIALIZABLE | 重复预约 |
| 批量增加副本 | book_copy(n), books | BATCH INSERT + UPDATE | SERIALIZABLE | 条码生成冲突 |

### 7.2 Sequelize 事务示例 — 借书操作

```javascript
// borrow.service.js
async borrowBooks(readerId, barcodes, operatorId) {
  const sequelize = this.db.sequelize;
  const transaction = await sequelize.transaction({
    isolationLevel: Transaction.ISOLATION_LEVELS.SERIALIZABLE
  });

  try {
    // 1. 锁定读者记录（悲观锁，防止并发）
    const reader = await Reader.findByPk(readerId, {
      lock: transaction.LOCK.UPDATE,
      transaction
    });
    if (!reader) throw new NotFoundError('读者不存在', 40301);
    if (reader.status !== 'active') throw new ForbiddenError('读者账户状态异常', 40302);
    if (reader.current_borrowed >= reader.borrow_limit) throw new ConflictError('已达借阅上限', 40502);

    // 2. 逐本校验
    const copies = [];
    for (const barcode of barcodes) {
      const copy = await BookCopy.findOne({
        where: { barcode },
        lock: transaction.LOCK.UPDATE,
        transaction
      });
      if (!copy) throw new NotFoundError(`条码 ${barcode} 不存在`, 40511);
      if (copy.status !== 'available') throw new ConflictError(`副本 ${barcode} 不可借`, 40408);
      copies.push(copy);
    }

    // 3. 读取系统配置
    const borrowDays = parseInt(await this.getConfig('borrow_duration_days', '30'), 10);
    const dueDate = dayjs().add(borrowDays, 'day').format('YYYY-MM-DD');

    // 4. 批量创建借阅记录
    const borrowRecords = [];
    for (const copy of copies) {
      const borrow = await Borrow.create({
        reader_id: readerId,
        copy_id: copy.id,
        operator_id: operatorId,
        borrow_date: new Date(),
        due_date: dueDate,
        status: 'borrowing',
        renew_count: 0
      }, { transaction });
      borrowRecords.push(borrow);
    }

    // 5. 批量更新副本状态
    const copyIds = copies.map(c => c.id);
    await BookCopy.update(
      { status: 'borrowed', updated_at: new Date() },
      { where: { id: copyIds }, transaction }
    );

    // 6. 更新读者借阅计数
    await Reader.update(
      { current_borrowed: reader.current_borrowed + barcodes.length, updated_at: new Date() },
      { where: { user_id: readerId }, transaction }
    );

    // 7. 更新图书可借副本数（使用 increment/decrement）
    const bookIds = [...new Set(copies.map(c => c.book_id))];
    for (const bookId of bookIds) {
      await Book.decrement(
        { available_copies: copies.filter(c => c.book_id === bookId).length },
        { where: { id: bookId }, transaction }
      );
    }

    // 8. 记录操作日志
    await AuditLog.create({
      user_id: operatorId,
      action: 'borrow.create',
      target_type: 'borrow',
      target_id: borrowRecords.map(b => b.id).join(','),
      detail: JSON.stringify({ reader_id: readerId, barcodes, count: barcodes.length }),
      ip_address: req.ip
    }, { transaction });

    // 9. 提交事务
    await transaction.commit();

    return borrowRecords.map(b => ({ ... }));
  } catch (error) {
    await transaction.rollback();
    throw error;
  }
}
```

### 8.2 读者冻结检查

**执行频率**：每日 00:05

```sql
-- 冻结逾期 30 天以上的读者
UPDATE readers
SET status = 'frozen', updated_at = datetime('now')
WHERE status = 'active'
  AND user_id IN (
    SELECT DISTINCT b.reader_id
    FROM borrow b
    WHERE b.status = 'overdue'
      AND CAST(julianday('now') - julianday(b.due_date) AS INTEGER) >= 30
  );
```

### 8.3 预约过期清理

**执行频率**：每日 00:10

```sql
UPDATE reserve
SET status = 'expired'
WHERE status = 'waiting'
  AND expire_date < date('now');
```

### 8.4 锁定账户自动解除

## 11. 性能优化建议

### 11.1 SQLite 参数调优

```sql
-- 数据库连接初始化时执行
PRAGMA journal_mode = WAL;           -- WAL 模式，提升并发读写
PRAGMA synchronous = NORMAL;         -- WAL 模式下 NORMAL 足够安全
PRAGMA cache_size = -8000;           -- 缓存 8MB
PRAGMA busy_timeout = 5000;          -- 锁等待 5 秒
PRAGMA temp_store = MEMORY;          -- 临时表存内存
PRAGMA mmap_size = 268435456;        -- 内存映射 256MB
PRAGMA foreign_keys = ON;            -- 开启外键约束
```

### 11.2 查询优化建议

1. **模糊搜索优化**：books 表 title 字段使用 `LIKE '%keyword%'` 不走索引。建议方案：
   - 使用 SQLite FTS5 全文搜索扩展建立全文索引
   - 或限制模糊搜索仅支持前缀匹配：`WHERE title LIKE 'keyword%'`

2. **大数据量分页**：audit_log 表数据量最大（200万+），使用 OFFSET 分页时深分页性能差。建议：
   - 使用游标分页：`WHERE created_at < ? ORDER BY created_at DESC LIMIT 20`
   - 或限制最大查询页数

3. **冗余字段维护**：`books.total_copies`/`available_copies` 必须在事务中更新，确保与 `book_copy` 表一致

4. **连接池配置**：SQLite 单写锁，Sequelize 连接池 max=1（写）、min=1（读）

### 11.3 数据归档建议

`audit_log` 表增长最快，建议：
- 保留最近 12 个月数据在主表
- 超过 12 个月的归档到 `audit_log_archive` 表
- 归档操作在业务低峰期执行，使用事务批量迁移

## 12. 测试用例建议

### 12.1 数据完整性测试

| 测试项 | 说明 |
|--------|------|
| PK 唯一性 | 向各表插入具有相同主键的数据，预期失败 |
| FK 约束 | 向 borrow 表插入不存在的 reader_id，预期失败 |
| CHECK 约束 | 向 readers 插入 current_borrowed > borrow_limit 的数据，预期失败 |
| UNIQUE 约束 | 向 books 插入重复 ISBN，预期失败 |
| 自增字段 | 连续插入 3 条，预期 id 递增 |
| 默认值 | 插入时不指定 role，预期为 'reader' |
| 非空约束 | 插入时 username 为 NULL，预期失败 |
